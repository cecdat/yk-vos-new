# YK-VOS 数据分析中台（新建）— 项目分析（v3）

> 本文件是**分析**（不产出业务代码）。目标：从零新建一套数据分析中台，替换现有 Python/Next.js 系统。
> 数据来源已变：旧=调 VOS Web API；新=**直接同步 VOS 数据库**。
> 新项目独立目录：`D:\github\yk-vos-new\`（旧仓库保持不动）。
> **数据模型唯一真值源：`vos3000_structure.sql`（VOS 服务器真实库结构），所有表设计以它为准。**
>
> **技术栈（已闭环确定）**：
> - 前端 **yudao-ui-admin-vben**（Vben5 + Vue3.5 + Ant Design Vue4 + Vite8 + Pinia + TS，pnpm monorepo，用 `apps/web-antd`）。
> - 后端 **YunaiV/ruoyi-vue-pro**（芋道 RuoYi-Vue Pro 单体：Spring Boot + MyBatis Plus + Redis + Quartz + 内置 RBAC/代码生成）。**二者同源原生配对，零适配**（见 §3.5）。

## 1. 数据来源根本变化（为什么不能再「重构」旧系统）

| 维度 | 旧（调 VOS Web API） | 新（直接同步 VOS 数据库） |
|---|---|---|
| 采集方式 | Celery 轮询 VOS Web API | 直连 VOS MySQL，SELECT / 批量导出 / CDC |
| 数据保真 | 仅 API 暴露的**计费级字段子集** | **全量原始字段**（codec / PDD / gateway / endreason …） |
| 实时性 | 受 API 限流、轮询间隔约束 | 取决于同步频率（CDC 近实时；批量分钟级） |
| 可靠性 | 依赖 VOS 接口稳定性与鉴权 | 依赖 VOS MySQL 只读访问与表结构稳定 |
| 吞吐成本 | 接口调用开销，单条/分页拉取 | 一次建连、整表批量拉取，吞吐高 |

**结论**：采集层与数据模型完全不同 → 重建优于在旧代码上重构。

## 2. VOS 生产库结构分析（vos3000_structure.sql — 唯一真值源）

- 共 **100+ 张表**，全部 `MyISAM` / `utf8_bin`。
- 核心域与中台映射：

| 域 | VOS 表 | 说明 |
|---|---|---|
| 话单（核心） | `e_cdr`（滚动当前表）+ `e_cdr_YYYYMMDD`（**51 个历史日表**，约 2026-01-18 ~ 2026-03-09） | 与 `e_cdr` **完全同构**，`flowno` 为唯一键 → 统一同步 + 去重 |
| 客户 | `e_customer`（含 `money` 余额 / `status` / `locktype` / `feerategroup_id`）、`e_customerdetail` | 标签/余额维度 |
| 网关 | `e_gatewaygroup` / `e_gatewaymapping` / `e_gatewayrouting` … | 网关标签/容量 |
| 费率 | `e_feerate` / `e_feerategroup` / `e_feeratebytime` / `e_feeratesection` | 费率解释 |
| VOS 自带预聚合 | `e_reportcustomerfee` / `e_reportgatewaymappingfee` / `e_reportcustomerio` … | 按日/客户/网关汇总，**仅作交叉校验，不自建依赖** |
| 其它模块（本期忽略） | `e_aas_cdr`(音频转写/关键词告警) / `e_ivr_cdr` / `e_axb_cdr` / `e_conference*` / `e_mbx` / `e_suite` … | 独立模块，后续视需 |
| VOS 自有用户 | `e_user` / `e_user_privilege` | **新中台不复用**，用 ruoyi-vue-pro 自带 RBAC |

- **`e_cdr` 字段全量 vs 现系统**：现系统只取计费级子集（`account` / `fee` / `agent_fee` / `suite_fee` / `call_level` 等）。`e_cdr` 多出：
  - 编解码与媒体：`callercodec` / `calleecodec`
  - 质量与时延：`callerpdd` / `calleepdd` / `holdtime`
  - 网关与软交换：`callergatewayid` / `calleegatewayid` / `callertogatewaye164` / `softswitchname` / `softswitchcallid` / `callercallid`
  - 挂断原因：`enddirection` / `endreason` / `sipreasonheader`（→ ASR / 失败分析）
  - 计费细化：`fee` / `tax` / `suitefee` / `incomefee` / `agentfee` / `agenttax` / `agentsuitefee` / `feetime` / `suitefeetime` / `billingmode` / `billingtype`
  - 客户与代理：`customeraccount` / `customername` / `agentaccount` / `agentname`
  - 时间：`starttime` / `stoptime` / `recordstarttime`（epoch 秒，bigint）
  - 其它：`transactionid` / `flownofirst` / `dynamicValue`(json) / `additional`(blob)

  → 这些字段让中台能做现系统缺失的多维分析（接通率、网关/编码维度、挂断原因分布、代理费用拆分）。

> **真值约束**：本中台所有 ODS / DWS / 维度表均由 `vos3000_structure.sql` 导出；任何设计文档的 CDR 模型若与之冲突，一律以 `vos3000_structure.sql` 为准。

## 3. 技术栈分析（已闭环）

### 3.1 后端：YunaiV/ruoyi-vue-pro（芋道 RuoYi-Vue Pro，已定）
- 即「芋道」平台的**单体版**仓库（作者 YunaiV），与前端 `yudao-ui-admin-vben` **同源同生态**；其官方 vben 前端演示 `dashboard-vben.yudao.iocoder.cn` 正是该前端仓库，证明**后端 + 前端原生配对、零 API 适配**。
- 技术栈（取 **`master-jdk17` 分支**，契合本机 JDK 17）：
  - **Spring Boot 3.5 + JDK 17**（另有 `master`=JDK8/SB2.7、`master-jdk25`=JDK25/SB4 分支，本机选 jdk17）。
  - **MyBatis Plus 3.5**（ORM）+ **MySQL**（5.7/8.0+）+ **Redis 7 + Redisson**（缓存 / Token）。
  - **Quartz 2.3** 定时任务（→ 批量同步兜底，与 Canal 实时互补）。
  - **Spring Security + Token + Redis** 鉴权，内置 RBAC。
- **模块化结构（cherry-pick 我们用到，其余不引入）**：
  - `yudao-server`：管理后台服务端（启动模块）
  - `yudao-module-system`：系统功能（**用户/角色/菜单 RBAC**、部门、字典 —— 三权分立直接满足）
  - `yudao-module-infra`：基础设施（**定时任务 Job**、Redis、文件等 —— 批量同步任务放这）
  - 不需的模块（mall / erp / crm / bpm / pay / im / ai / iot / mp / report 等）**直接不引入**，保持轻量。
- **关键收益（旧系统最大短板被补齐）**：
  - RBAC 开箱即用（用户/角色/菜单 + 按钮级权限 + 数据范围权限）→ 三权分立 admin/ops/finance/ops_mgr 直接落地，无需手搓。
  - 代码生成器：一键生成 Java/Vue/SQL → `e_customer`/`e_gatewaymapping`/`e_feerate` 等维度表 CRUD 快搭。
  - Quartz：历史 51 日表批量回溯 + 实时增量轮询兜底。

### 3.2 前端：yudao-ui-admin-vben（已定，与后端同源）
- 基于 **Vben5（vue-vben-admin v5.7）** + **Vue3.5** + **Ant Design Vue4** + Vite8 + Pinia + TS6，pnpm monorepo。
- 多套 UI 模板并存（`apps/web-antd` 我们用这个、`apps/web-ele`、`apps/web-naive`…），公共包 `packages/`。
- **它正是 ruoyi-vue-pro 的 vben(AntDV) 前端版**，与 §3.1 后端零适配。
- 工具链：需 **pnpm** + Node ≥ 20（本机 Node 22 ✅）。

### 3.3 数据层
- **ClickHouse**：原始话单事实表 `vos_cdr_ods`（ODS）+ DWS 汇总（如 `vos_cdr_daily_customer`）。承载海量 CDR 分析与看板查询。DDL 见 `clickhouse/ods/01_ods_vos.sql`（已按真实 `e_cdr` 镜像建立）。
- **MySQL**：① ruoyi-vue-pro 自身元数据（RBAC/字典/任务）；② **VOS 维度表镜像**（`e_customer` / `e_gatewaymapping` / `e_feerate`，用于标签/联表，避免每次查 ClickHouse 做维度 join）。

### 3.4 同步链路：批量优先 + 可选 Canal
- **历史 51 日表 + 滚动 `e_cdr`：必走批量 SELECT** 回溯入 ClickHouse（Canal 无法回放历史，且 VOS 表为 MyISAM）。
- **实时增量**：先试 Canal（binlog=ROW）；不可用则降级按 `starttime > last_max` 批量增量。采集源可插拔（`CanalSource` / `BatchSource`）。详见 §5。

### 3.5 前后端配对冲突——已消解 ✅
- 上一轮曾误判「yudao-ui-admin-vben 只配对 yudao-server、与 RuoYi 冲突」。实测：`YunaiV/ruoyi-vue-pro` 就是**芋道单体仓库**，而 `yudao-ui-admin-vben` 是其官方 vben(AntDV) 前端（演示 `dashboard-vben.yudao.iocoder.cn`）。**二者本就是一套，零适配。**
- 因此「方案 A(芋道)/方案 B(RuoYi 高适配)」二选一不再成立——你给的 `ruoyi-vue-pro` 后端 + `yudao-ui-admin-vben` 前端，**正是该生态的标准组合**。
- 旧 `RuoYi-Vue-master.zip`（Vue2+Element UI 老版）可不用了，ruoyi-vue-pro 是其现代化继任（Vue3 + SB3.5 + 模块化）。

## 4. 整体架构（C4 容器级）

```
[VOS MySQL 只读/从库]
   │  批量回溯(历史51日表)        │  CDC: Canal→Kafka(可选,实时)
   └───────────────┬──────────────┘
                      ▼
        [ClickHouse]  ODS(vos_cdr_ods) → DWS(汇总) → ADS(看板)
                      ▲
        [MySQL]       ruoyi-vue-pro 元数据(RBAC) + VOS 维度镜像(e_customer/gateway/feerate)
                      ▲
        [后端: 芋道 RuoYi-Vue Pro / ruoyi-vue-pro]  RBAC / 代码生成 / Quartz / 业务 API
                      ▲
        [前端: yudao-ui-admin-vben (apps/web-antd)]  AntDV + TS + Pinia + Vben5
```

## 5. 关键技术风险与调和：Canal vs MyISAM

- Canal 需 MySQL `binlog=ROW`；但 **VOS3000 表是 `MyISAM`**，且 **Canal 无法回放历史**。
- 调和方案（采集源可插拔）：
  - **历史 51 日表 + 滚动 `e_cdr`：必走批量 SELECT** 回溯入 ClickHouse（Canal 做不到历史回放）。
  - **实时增量**：先试 Canal；若 VOS binlog 不可用 / 非 ROW → 降级为按 `starttime > last_max` 的批量增量。
  - 抽象 `CdrSource` 接口，`CanalSource` 与 `BatchSource` 各自实现 → 不把赌注全押 Canal。
- 回溯清单见 `scripts/cdr_backfill_manifest.json`（51 日表 + 滚动表，去重键 `flowno`）。

## 8. 构建顺序（全量但分阶段可验）

1. **基础设施**：新目录骨架 + ClickHouse DDL 执行 + MySQL（ruoyi-vue-pro）初始化。
2. **后端 ruoyi-vue-pro 起底**：RBAC（module-system）+ Quartz（module-infra）+ 代码生成。
3. **同步层**：`BatchSource` 历史回溯（51 日表）→ ClickHouse；再视情况接 `CanalSource`（实时）。
4. **业务模块**：客户/网关/费率（代码生成）+ 看板/统计（查 ClickHouse）。
5. **前端**：yudao-ui-admin-vben 登录/权限菜单 + 仪表盘/统计/客户/网关/话单/财务页面。
6. **其余模块补齐**：缓存/设置/对接账户明细/ivr/axb 等。

## 9. 设计文档纠偏（你那份《VOS3000数据分析中台项目设计文档.md》）

- 设计文档的 CDR 模型写的是 **SIP 级**（`call_id` / `sip_status` / `codec` / `Q.850`）——与真实 VOS 库不符；真实 `e_cdr` 是**计费级**（`account` / `fee` / `agent_fee` …）。新中台以真实 `e_cdr`（即 `vos3000_structure.sql`）为准。

- 设计文档采集写的是 **Canal 主链路** —— 实际需批量优先（Canal 因 MyISAM 无法历史回放）。
- 设计文档后端栈未明确指定 —— 现已定为 **ruoyi-vue-pro（芋道单体）**，其原生提供 RBAC/代码生成/Quartz，无需手搓。
