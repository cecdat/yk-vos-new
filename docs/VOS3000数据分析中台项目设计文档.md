# **VOS3000 数据分析中台（话务数据中台）项目设计文档**

本设计文档旨在指导开发团队及 AI 开发助手（Codex/Cursor/Claude Code）进行系统构建。文档详细定义了系统架构、数据流向、双数据库设计（PostgreSQL \+ ClickHouse）以及基于角色（运维、财务、运营）的核心业务实现。

## **1\. 系统架构与数据流向设计**

系统采用**基于 Binlog 的实时流式架构**，以保障在电信级高并发（日千万级话单）下的低延迟、低耦合与高可用性。

### **1.1 核心技术拓扑**

`[ VOS3000 MySQL (生产源) ]`   
       `│`   
       `▼ (1. 实时解析 Binlog: 仅监听 cdr_.* 表)`  
  `[ Canal Deployer ]`   
       `│`   
       `▼ (2. 投递 JSON 消息，按主叫号码 caller_num 分区)`  
  `[ Apache Kafka (Topic: vos_cdr_raw) ]`   
       `│`   
       `▼ (3. RuoYi-Vue3 后端消费：攒批缓冲队列 BlockingQueue)`  
  `[ Java Consumer (Spring Boot) ]`   
       `│`  
       `├─ (4a. 批量写入，5000条或2秒/批) ──────► [ ClickHouse (分析库: ODS/DWS) ]`  
       `│                                            └─► 支撑：运维大屏、运营决策、合规质检`  
       `│`  
       `└─ (4b. 业务扣费流：单条事务性扣费) ────► [ PostgreSQL (核心主库: master) ]`  
                                                    `└─► 支撑：账户余额、财务账单、套餐额度`

### **1.2 关键组件技术栈**

* **基础后台框架**：RuoYi-Vue3 (Spring Boot 3 \+ Vue 3 \+ Element Plus)  
* **核心主数据库**：PostgreSQL 15+ (管理用户、权限、配置、账户余额、财务流水)  
* **分析列式数据库**：ClickHouse 23.x+ (存储海量 ODS 话单及 DWS 聚合数据)  
* **Binlog 采集器**：Alibaba Canal  
* **消息缓冲中间件**：Apache Kafka (保证削峰填谷，避免冲垮 ClickHouse)

## **2\. 数据库建模与设计 (Database Schema)**

### **2.1 分析库：ClickHouse ODS 话单表设计**

在 ClickHouse 中建立 ods 库及本地明细表。使用 MergeTree 引擎，采用低基数（LowCardinality）与专用 IP 类型优化存储结构。  
`CREATE DATABASE IF NOT EXISTS ods;`

`CREATE TABLE ods.t_vos_cdr_local ON CLUSTER default_cluster`  
`(`  
    `-- 1. 呼叫唯一标识`  
    `` `call_id` String COMMENT '呼叫唯一ID', ``  
    `` `session_id` String COMMENT '会话ID', ``  
      
    `-- 2. 时间维度`  
    `` `setup_time` DateTime64(3) COMMENT '呼叫发起时间', ``  
    `` `alert_time` DateTime64(3) COMMENT '振铃时间', ``  
    `` `connect_time` DateTime64(3) COMMENT '接通时间', ``  
    `` `stop_time` DateTime64(3) COMMENT '挂断时间', ``  
      
    `-- 3. 通话时长指标`  
    `` `duration` UInt32 COMMENT '总时长(秒)', ``  
    `` `billable_duration` UInt32 COMMENT '计费时长(秒)', ``  
    `` `pdd` UInt32 COMMENT '拨号后时延(ms)', ``  
      
    `-- 4. 呼叫源与目的`  
    `` `caller_num` String COMMENT '原始主叫', ``  
    `` `callee_num` String COMMENT '原始被叫', ``  
    `` `display_num` String COMMENT '显示主叫', ``  
      
    `-- 5. 账户与网关 (高频过滤维度采用低基数压缩)`  
    `` `account_id` LowCardinality(String) COMMENT '客户账户ID', ``  
    `` `incoming_gw` LowCardinality(String) COMMENT '接入网关ID', ``  
    `` `outgoing_gw` LowCardinality(String) COMMENT '落地网关ID', ``  
    `` `route_prefix` LowCardinality(String) COMMENT '路由前缀', ``  
      
    `-- 6. 网络与媒体 (采用IPv4及低基数类型)`  
    `` `caller_ip` IPv4 COMMENT '主叫IP', ``  
    `` `callee_ip` IPv4 COMMENT '被叫IP', ``  
    `` `caller_codec` LowCardinality(String) COMMENT '主叫编码', ``  
    `` `callee_codec` LowCardinality(String) COMMENT '被叫编码', ``  
      
    `-- 7. 挂断状态与原因`  
    `` `disconnect_reason` UInt16 COMMENT 'Q.850 挂断原因值', ``  
    `` `sip_status` UInt16 COMMENT 'SIP 状态码', ``  
      
    `-- 8. 财务财务数据`  
    `` `customer_rate` Decimal(10, 4) COMMENT '客户费率', ``  
    `` `customer_fee` Decimal(10, 4) COMMENT '客户消费金额', ``  
    `` `provider_rate` Decimal(10, 4) COMMENT '落地费率', ``  
    `` `provider_fee` Decimal(10, 4) COMMENT '落地成本金额', ``  
      
    `-- 9. 写入审计`  
    `` `sync_time` DateTime DEFAULT now() COMMENT '中台同步时间' ``  
`)`  
`ENGINE = MergeTree()`  
`PARTITION BY toYYYYMMDD(stop_time)`  
`ORDER BY (toYYYYMMDD(stop_time), account_id, outgoing_gw, incoming_gw, sip_status)`  
`SETTINGS index_granularity = 8192;`

### **2.2 核心主库：PostgreSQL 客户余额与配置表**

在 PostgreSQL 中存放强事务性、强关联性的业务配置数据。  
`-- 客户账户及余额表`  
`CREATE TABLE public.e_customer (`  
    `account_id VARCHAR(64) PRIMARY KEY,`  
    `name VARCHAR(128) NOT NULL,`  
    `money DECIMAL(15, 4) NOT NULL DEFAULT 0.0000,           -- 账户余额（高精度）`  
    `credit_limit DECIMAL(15, 4) NOT NULL DEFAULT 0.0000,    -- 信用额度`  
    `status INT NOT NULL DEFAULT 0,                          -- 0: 正常, 1: 锁定`  
    `today_consumption DECIMAL(15, 4) NOT NULL DEFAULT 0.0, -- 今日消费`  
    `update_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP`  
`);`

`-- 账户充值历史`  
`CREATE TABLE public.e_payhistory (`  
    `id BIGSERIAL PRIMARY KEY,`  
    `account_id VARCHAR(64) REFERENCES e_customer(account_id),`  
    `amount DECIMAL(15, 4) NOT NULL,`  
    `pay_time TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,`  
    `operator_id BIGINT,`  
    `remark TEXT`  
`);`

## **4\. Kafka 攒批消费与性能规约（核心代码控制）**

为了防止 ClickHouse 因小批次高频写入引发 Too many parts 崩溃，Java 消费端必须实现 **“内存双重检查攒批写入器”**。  
`@Component`  
`@Slf4j`  
`public class CdrRealtimeConsumer {`

    `// 内存缓冲区队列`  
    `private final BlockingQueue<VosCdr> queue = new LinkedBlockingQueue<>(100000);`  
    `private static final int BATCH_SIZE = 5000; // 达到5000条触发写入`  
      
    `@Autowired`  
    `private ClickHouseCdrMapper clickHouseCdrMapper;`

    `/**`  
     `* 1. 消费 Kafka 话单消息并压入内存`  
     `*/`  
    `@KafkaListener(topics = "vos_cdr_raw", groupId = "ruoyi-ta-consumers")`  
    `public void onMessage(ConsumerRecord<String, String> record) {`  
        `try {`  
            `VosCdr cdr = parseCanalJsonToCdr(record.value());`  
            `if (cdr != null) {`  
                `queue.put(cdr);`  
            `}`  
            `if (queue.size() >= BATCH_SIZE) {`  
                `triggerFlush();`  
            `}`  
        `} catch (Exception e) {`  
            `log.error("消费消息处理异常: ", e);`  
        `}`  
    `}`

    `/**`  
     `* 2. 定时器补偿：每 2 秒强制刷盘一次，防止低话务量时数据延迟`  
     `*/`  
    `@Scheduled(fixedRate = 2000)`  
    `public void scheduledFlush() {`  
        `if (!queue.isEmpty()) {`  
            `triggerFlush();`  
        `}`  
    `}`

    `/**`  
     `* 3. 核心刷盘动作 (带同步锁避免多线程冲突)`  
     `*/`  
    `private synchronized void triggerFlush() {`  
        `if (queue.isEmpty()) return;`  
        `List<VosCdr> batchList = new ArrayList<>();`  
        `queue.drainTo(batchList, BATCH_SIZE);`  
        `try {`  
            `long startTime = System.currentTimeMillis();`  
            `clickHouseCdrMapper.batchInsert(batchList);`  
            `log.info("Successfully flushed {} CDR records to ClickHouse in {} ms", batchList.size(), (System.currentTimeMillis() - startTime));`  
        `} catch (Exception e) {`  
            `log.error("Failed to batch insert into ClickHouse", e);`  
            `// TODO: 此处应增加 DLQ(死信队列) 写入或本地磁盘缓存文件规避丢单`  
        `}`  
    `}`  
`}`

## **5\. 多角色业务功能场景与权限控制 (RBAC)**

基于 RuoYi-Vue3 原生的 Spring Security 角色机制，将菜单路由及按钮权限进行三权分立隔离：

### **5.1 运维角色视图 (role\_ops / 权限符: ta:ops:\*)**

* **核心关注指标**：并发（Concurrent Calls）、CAPS、PDD（拨号延时）、SIP 状态码。  
* **技术看板核心 SQL 逻辑（ClickHouse 极速过滤）**：

`-- 按分钟统计过去 1 小时内全网 SIP 错误分类及占比`  
`SELECT`   
    `toStartOfMinute(setup_time) AS event_time,`  
    `sip_status,`  
    `count(1) AS count_num`  
`FROM ods.t_vos_cdr_local`  
`WHERE setup_time >= now() - INTERVAL 1 HOUR AND sip_status != 200`  
`GROUP BY event_time, sip_status`  
`ORDER BY event_time DESC, count_num DESC;`

### **5.2 财务角色视图 (role\_finance / 权限符: ta:finance:\*)**

* **核心关注指标**：充值流水（pay\_history）、今日毛利（客户消费 \- 落地成本）、客户信用额度（PG 数据）、欠费预警。  
* **自助对账核算 SQL (ClickHouse & PG 联动逻辑)**：

`-- 实时按客户维度统计今天毛利额（财务看板首屏，毫秒级响应）`  
`SELECT`   
    `account_id,`  
    `sum(customer_fee) AS total_revenue,`  
    `sum(provider_fee) AS total_cost,`  
    `(total_revenue - total_cost) AS gross_profit,`  
    `round((gross_profit / total_revenue) * 100, 2) AS profit_margin_pct`  
`FROM ods.t_vos_cdr_local`  
`WHERE toYYYYMMDD(stop_time) = toYYYYMMDD(now())`  
`GROUP BY account_id`  
`ORDER BY gross_profit DESC;`

### **5.3 运营角色视图 (role\_ops\_mgr / 权限符: ta:operation:\*)**

* **核心关注指标**：ASR（接通率）、ACD（平均通话时长）、闪断率、敏感词/涉诈合规拦截。  
* **ASR / ACD 指标监控 SQL**：

`-- 按落地网关统计过去 10 分钟 ASR 和 ACD，低于阈值触发运营黄色警报`  
`SELECT`   
    `outgoing_gw,`  
    `count(1) AS total_attempts,`  
    `sum(case when sip_status = 200 then 1 else 0 end) as connected_calls,`  
    `round(connected_calls / total_attempts * 100, 2) as asr,`  
    `round(sum(duration) / connected_calls, 1) as acd`  
`FROM ods.t_vos_cdr_local`  
`WHERE setup_time >= now() - INTERVAL 10 MINUTE`  
`GROUP BY outgoing_gw`  
`HAVING total_attempts > 100;`

## **6\. 二次开发实操避坑指南（Codex 重点解析）**

1. **ClickHouse 数据不可变性约束**：  
   * ClickHouse **不支持** 标准的带有大范围事务的 UPDATE 或 DELETE 操作。  
   * AI 助手在重构话单处理业务时，禁止生成任何针对 ods.t\_vos\_cdr\_local 的修改代码。若话单必须修正，必须采用 ReplacingMergeTree 引擎通过版本号更新，或者直接在数仓轻度汇总层进行数据重构。  
2. **PostgreSQL 超时与死锁预防**：  
   * 当财务进行高并发批量扣费时，必须优先锁定 account\_id，防止多条扣费线程引起 PostgreSQL 死锁。  
   * 所有 PG 的财务读写代码，必须带有 @Transactional(rollbackFor \= Exception.class)，且执行时间不可超过 3 秒。  
3. **大宽表空字段处理规则**：  
   * VOS 话单中存在大量未接通呼叫（如无接通时间、无结束原因）。在 Java 写入 ClickHouse 前，禁止将这些字段设为 null，而应填充默认初始值（例如：connect\_time 初始化为 1970-01-01 00:00:00，sip\_status 设为 0），以避免 ClickHouse 的 Nullable 字段造成的 1.5 倍以上存储空间膨胀和查询降速。
