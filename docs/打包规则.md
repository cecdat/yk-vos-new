# YK-VOS 打包规则文档

> 适用范围：正式项目 `yk-vos-new`（VOS 数据中台，栈 = yudao/Java + vben/Vue + Go agent + ClickHouse 数据底座）。
> 本文档只讲**构建 + 打包**（build → package），部署见同目录《部署手册.md》，两者互补、不重复。
> 与参考项目 `yk-vos`（走 VOS API，数据不准）无任何关系，请勿混用其结论。

---

## 0. 一句话流程

```
后端改动 →  Maven 构建 yudao-server.jar（必须带 -am，且在 backend/ 下跑）
前端改动 →  pnpm 构建 web-ele 的 dist
两者就绪 →  bash release/package_server.sh dev <版本号>  →  ykvos-server-dev-<版本>.tar.gz
```

---

## 1. 环境先决条件（错一个就卡住）

| 项 | 规则 | 错误做法 |
|---|---|---|
| **JDK** | 强制 **JDK17**。`tools/mvn` 内部已锁 `/c/Program Files/Eclipse Adoptium/jdk-17.0.19.10-hotspot`，不要改全局 `JAVA_HOME` | 用默认 JDK11 构建 yudao（master-jdk17 分支）会编译失败 |
| **Maven** | 一律用 **`bash tools/mvn`** 包装脚本 | 直接 `mvn` 会因 Git Bash 的 `MSYS_NO_PATHCONV=1` 把 `/d/...` 原样传给 `java.exe`，报「找不到或无法加载主类 org.codehaus.plexus.classworlds.launcher.Launcher」 |
| **Node/pnpm** | 本机 corepack 损坏，前端一律 `npx -y pnpm@11 install` / `npx -y pnpm@11 --filter @vben/web-ele build` | 全局 `pnpm install`（corepack 坏，装不上或装错版本） |
| **内存** | 构建前 `export JAVA_TOOL_OPTIONS=-Xmx2g` 防 OOM | 默认堆太小，大模块编译 OOM 中断 |

## 2. 后端构建规则（⚠️ 最重要，踩坑重灾区）

### 2.1 必须在 `backend/` 目录下执行
`tools/mvn` 脚本把 `maven.multiModuleProjectDirectory` 设为 `$(pwd)`（当前目录）。
在**仓库根目录**跑会报：

```
[ERROR] Could not find the selected project in the reactor: yudao-server
```

✅ 正确：
```bash
cd /d/github/yk-vos-new/backend
export JAVA_TOOL_OPTIONS=-Xmx2g
bash /d/github/yk-vos-new/tools/mvn -pl yudao-server -am package -DskipTests
```

### 2.2 参数含义（一个都不能省）
- `-pl yudao-server`：只构建这个模块（最终运行物）。
- **`-am`（also-make）**：连带构建它的依赖模块（如 `yudao-module-vos`、`yudao-framework/*`）。
  - ⚠️ **改了 `yudao-module-*` 或 `yudao-framework/*` 的代码，必须带 `-am`**，否则只重打 `yudao-server`、不重编依赖模块 → **你的修复根本进不了 jar**（这正是本次 1.0.3 第一次构建失手的坑）。
- `-DskipTests`：跳过单测（本机跑测试慢且易 OOM）。

### 2.3 产物
- `backend/yudao-server/target/yudao-server.jar`（Spring Boot 胖 jar，依赖全在 `BOOT-INF/lib/`）。
- 若怀疑增量编译取了旧 class，改用 `clean package` 强制全量重编（更慢但保险）。

### 2.4 🔴 构建后必做：验证 jar 内确实含本次改动
胖 jar 是「jar 套 jar」，依赖模块的 class 在 `BOOT-INF/lib/<module>-*.jar` 里，**直接看 `target/yudao-server.jar` 的 `BOOT-INF/classes/` 是查不到的**（那只有 yudao-server 自身类）。必须用下面命令校验：

```bash
# 抽取嵌套的 module-vos jar（按实际版本号替换）
unzip -p backend/yudao-server/target/yudao-server.jar \
  "BOOT-INF/lib/yudao-module-vos-2026.06-SNAPSHOT.jar" > /tmp/mv.jar

# 查某个被改类的修复标识（例：本次 Kafka 预热修复）
unzip -p /tmp/mv.jar \
  "cn/iocoder/yudao/module/vos/framework/kafka/AgentCommandProducer.class" \
  | grep -c "warmUpProducer"
# 输出 1 = 含修复；输出 0 = 没打进去，必须重 build（clean package）
```
> 经验：本次曾出现「`mvn package` 日志 BUILD SUCCESS、但嵌套 jar 仍是旧副本、fat jar 时间戳未更新」的假象。凡改了 module 代码，**打包后一律用上面命令验证一次**，别信日志。

## 3. 前端构建规则

```bash
cd /d/github/yk-vos-new/frontend
export CI=true                       # 跳过 lefthook 等非必要步骤
npx -y pnpm@11 install --frozen-lockfile || npx -y pnpm@11 install
npx -y pnpm@11 --filter @vben/web-ele build
```
或直接用封装脚本（等价，已含 `CI=true`）：
```bash
bash /d/github/yk-vos-new/scripts/build_web_ele.sh
```

### 3.1 前端配置铁律
- 生产构建读 `frontend/apps/web-ele/.env.production`，其中 **`VITE_GLOB_API_URL=/admin-api`** 必须正确（前端同源反代，靠 nginx 把 `/admin-api` → `backend:48080`）。
- 若后端 API 前缀变了，前端 `.env.production` 必须同步，否则所有接口 404。
- 产物：`frontend/apps/web-ele/dist/`。

## 4. 一键合并打包

```bash
cd /d/github/yk-vos-new
bash release/package_server.sh dev 1.0.3      # 环境[dev|prod]  版本号
```

### 4.1 脚本行为（自动判缺、自动补建）
1. `backend/yudao-server/target/yudao-server.jar` 缺失 → 自动 `cd backend && tools/mvn -pl yudao-server -am package -DskipTests`。
2. `frontend/apps/web-ele/dist` 缺失或为空 → 自动 `scripts/build_web_ele.sh`。
3. 拷贝 `backend/sql/mysql/*.sql` 进包。
4. `tar -czf` 整个 `release/<env>/`，**排除 `ykvos-server-*.tar.gz` 自身旧包**，但**保留 `agent/ykvos-agent-*.tar.gz`**（当前组件，须随包分发）。

### 4.2 产物
`release/<env>/ykvos-server-<env>-<ver>.tar.gz`（如 `release/dev/ykvos-server-dev-1.0.3.tar.gz`）。

### 4.3 版本号规则
- 每次发版递增（1.0.1 → 1.0.2 → 1.0.3），**不要覆盖旧包文件名**——脚本已自动排除旧包防嵌套进新包。
- 包内 `README.md` / `部署手册.md` 提到的版本号记得同步更新。

## 5. 命名与库名约定（改了会出事）

| 资源 | 约定值 | 改时同步点 |
|---|---|---|
| **MySQL 业务库** | `ykvos`（非 yudao 原 `ruoyi-vue-pro`） | `docker-compose.yaml` 的 `MYSQL_DATABASE` + 两处 JDBC URL；SQL 文件名；`init/*.sh` 里的 `MYSQL_DB` |
| **ClickHouse ODS 库** | `ykvos_ch`（非 `ykvos`，避免与 MySQL 同名混淆） | `docker-compose.yaml` 的 `CLICKHOUSE_DB`；`application.yaml` 的 `jdbc-url: jdbc:clickhouse://clickhouse:8123/ykvos_ch` |
| **配置命名空间** | `ykvos.clickhouse.*`（key 前缀） | 改名库名时**不要顺手改 key 前缀**，只改库名值 |

⚠️ **SQL 文件里不要写死 `CREATE DATABASE`/`USE` 旧库名**：
- 已 grep 确认 `ykvos.sql` 里那两处 `ruoyi-vue-pro` 只是 **S3 桶名配置**，无害，**勿误改**。
- 建表 SQL 不带库名前缀，靠 `MYSQL_DATABASE` 决定落哪个库。

## 6. 部署侧强制规则（与打包强相关，必读）

> 详见《部署手册.md》，此处只列「因打包/初始化顺序踩过的坑」。

1. **`docker compose up -d` 必须带 `mysql-init` 服务**（完整列表：`mysql redis backend frontend clickhouse kafka monitor mysql-init`）。漏掉它，MySQL 自愈脚本不跑，库变空壳。
2. **MySQL 是 bind mount `./data/mysql`，`docker compose down` 不删数据卷**。要全量重导必须：
   ```bash
   docker compose down
   rm -rf ./data/mysql ./data/clickhouse ./data/kafka
   docker compose up -d
   ```
3. **含 `$` 的密码哈希（如 BCrypt）必须用 `<<'SQL'` 引号 heredoc 写入**：
   ```bash
   docker compose exec -T mysql mysql -uroot -p123456 ykvos <<'SQL'
   UPDATE system_users SET password='$2a$04$...' WHERE username='admin';
   SQL
   ```
   ❌ 绝不能用 `bash -c " ... SET password='\$2a\$04\$...' ..."` 双引号——bash 会把 `$2`/`$0`/`$04` 当变量展开，写坏哈希导致密码永远错。
4. **全量表数 = 63**（ykvos.sql 48 + quartz.sql 11 + vos.sql 4），不是「几百张」，别误判缺表。
5. 默认管理员 `admin` / `admin123`（yudao 内置 seed，首次登录后改密）。

## 7. 🔴 代码变更后、打包前必查清单

凡改动涉及「在异步 / ForkJoinPool / 定时任务线程上首次使用的组件」，需先确认代码侧已规避类加载器坑，否则打出来的包运行时会 `ClassNotFoundException`。

- ⚠️ **Kafka Producer 预热规则**：yudao 框架的 `TenantKafkaEnvironmentPostProcessor` 会给**所有 producer** 自动注入租户拦截器 `TenantKafkaProducerInterceptor`（在 `BOOT-INF/lib` 里）。producer 懒构造，若首次 `send()` 发生在定时/异步线程（上下文类加载器是系统 `AppClassLoader`，看不到 `BOOT-INF/lib`），会报 `ClassNotFoundException`。
  - **规则**：所有在此类线程首用的 Kafka producer，**必须在主线程预热**（`@PostConstruct` 里调 `kafkaTemplate.getProducerFactory().createProducer()`，单例缓存后后续线程复用），并在 `send()` 时强制 `Thread.currentThread().setContextClassLoader(getClass().getClassLoader())` 兜底。
  - 参考已实现：`AgentCommandProducer.warmUpProducer()`。
