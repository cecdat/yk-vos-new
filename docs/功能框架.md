# YK-VOS 数据分析中台 — 整体功能框架

> 本文档聚焦**顶层功能框架**：系统定位、模块域划分、菜单树、分期范围、模块依赖与边界。
> 详细的表结构 / API / 前端组件设计见 `docs/IVS平台复刻方案.md`（落地方案）。
> 数据模型唯一真值源：`vos3000_structure.sql`（VOS 服务器真实库结构）。

---

## 一、系统定位

| 项 | 说明 |
|---|------|
| 系统名 | YK-VOS 数据分析中台（代号 IVS 复刻） |
| 核心价值 | 直连 VOS 数据库 + ClickHouse 预聚合，做 VoIP 运营全链路管理（采集→客户→线路→财务→风控→分析） |
| 相对原版优势 | 全字段保真采集、亿级话单秒级分析、模块化可插拔、规范 RBAC |
| 数据来源 | VOS3000 MySQL（只读/从库）直连，替代原版 Web API 方式 |
| 技术栈 | 后端 ruoyi-vue-pro（SB3.5+JDK17）/ 前端 yudao-ui-admin-vben（Vben5+Vue3.5+AntDV4）/ 分析 ClickHouse |

---

## 二、功能域划分（4 + 1）

系统按业务职责划分为 **4 个业务域 + 1 个基础域（系统管理）**：

| 域 | 职责 | 包含模块 |
|----|------|---------|
| **0. 系统管理** | 基础能力（ruoyi 内置，零开发） | 用户 / 角色 / 菜单 / 部门 / 字典 / 岗位 / 操作日志 / 登录日志 |
| **1. 对接与采集** | 数据源接入与同步 | VOS 管理 / 话单同步 / 平台对接 / 黑名单对接 / 号码写回对接 / ASR 对接 |
| **2. 业务资源** | 运营底层资源 | 供应商入库 / 线路仓库 / 号码管理 / 号码标记 |
| **3. 客户运营** | 客户全生命周期 | 客户管理 / 财务管理 / 风控管理 / 统计报表 / ASR 质检 / 预警管理 / 黑名单管理 |
| **4. 路由策略** | 呼叫路由配置 | 主叫管理 / 被叫管理 / 碳通管理 |

## 五、Phase 1 MVP 范围（建议先交付）

**目标：** 跑通「多 VOS 实例配置 → 话单同步 → 基础业务管理」最小可用链路。

**包含 6 个模块：**
1. **VOS 管理** — 多实例 CRUD（截图当前页）
2. **话单同步** — BatchSource + Quartz 定时 + 同步日志
3. **客户管理** — e_customer 镜像列表 / 余额 / 费率组
4. **供应商入库** — 供应商 CRUD
5. **线路仓库** — e_gatewaymapping 镜像列表 / 状态
6. **黑名单管理** — 号码黑名单 CRUD

**配套支撑（非独立菜单，但为 MVP 必需）：**
- ClickHouse ODS 层（vos_cdr_ods 等 4 张表，DDL 已就绪）
- BatchSource 同步核心（JDBC 连 VOS MySQL → 批量写 ClickHouse）
- 系统管理 RBAC（ruoyi 内置）

**明确不包含（留待 Phase 2/3）：**
- 所有统计报表 / ECharts 图表
- 财务充值 / 风控规则 / 预警
- 路由策略（主叫 / 被叫 / 碳通）
- 外部对接（平台 / 黑名单 / 写回 / ASR）

> MVP 完成后，系统即具备原版 IVS 最核心的「**配置数据源 + 同步话单 + 管客户/线路/黑名单**」能力，可作为后续迭代的稳固地基。

## 六、模块依赖关系

```
[系统管理 P0] ─── 所有业务模块依赖（RBAC / 菜单 / 字典）
      │
      ▼
[VOS 管理 P1] ───> [话单同步 P1] ───> [ClickHouse ODS]
                                    │
                                    ▼
                          [统计报表 P2]（依赖 ODS/DWS）
                          [风控管理 P2]（依赖 ODS）
                          [财务管理 P2]（依赖 ODS 费用字段）
      │
      ▼
[线路仓库 P1] ───> [主叫/被叫/碳通 路由 P2]（规则绑定网关）
      │
      ▼
[客户管理 P1] ───> [财务充值 P2] / [风控事件 P2] / [预警 P2]
      │
      ▼
[号码管理 P2] ───> [号码标记 P2] ───> [号码写回对接 P2]
      │
      ▼
[ASR 质检 P3] ───> [ASR 对接 P3]
```

**关键路径：** 系统管理 → VOS 管理 → 话单同步 → ClickHouse → 上层分析（报表/风控/财务）。

## 七、边界与排除项（明确不做的）

| 项 | 说明 |
|----|------|
| VOS 自有用户体系（e_user） | 不复用，统一用 ruoyi RBAC |
| VOS 预聚合表（e_report*） | 仅交叉校验，不自建依赖（我们用 ClickHouse 自建更细的 DWS） |
| IVR / AXB / 会议 / 邮箱（e_ivr_cdr / e_axb_cdr / e_conference* / e_mbx） | 独立模块，本期不纳入 |
| 实际路由执行 | 路由规则在本中台配置 + 可视化，执行仍在 VOS3000 服务器端 |
| ASR 服务采购 | Phase 3 独立预算，按需启动，不影响 MVP |

## 九、下一步建议

框架梳理对齐后，建议按以下顺序推进：

1. **确认框架**：核对本文档的域划分 / 菜单树 / MVP 范围是否与预期一致（尤其 Phase 1 的 6 个模块取舍）。
2. **前端骨架**：在 yudao-ui-admin-vben 中按菜单树建立**路由 + 空页面占位**（先不填功能），让整体框架可见、可导航。
3. **后端模块**：创建 `yudao-module-vos`，按 MVP 6 表建 DDL + 用代码生成器出 CRUD 骨架。
4. **打通首页面**：以「VOS 管理」为第一个完整功能，验证前后端联调全链路。
5. **同步核心**：实现 BatchSource 把 e_cdr 拉入 ClickHouse，完成 MVP 最关键的一环。

> 框架层确认后，落地细节（每张表字段、每个 API 路径、每个图表配置）均可直接参照 `docs/IVS平台复刻方案.md` 执行，无需再次评审。
