# 受控历史话单回填 — 设计文档

> 状态：设计定稿（未实施）
> 适用范围：`yk-vos-new`（正式项目，直接读 VOS 服务器本机数据库取数；栈 yudao + MySQL + vben + ClickHouse 数据底座）。
> 关联规格：`docs/VOS同步Agent规格.md`（agent 基础机制）。

---

## 1. 背景与目标

### 1.1 现状（来自 `agent/` 实际代码）

- agent 是 **纯 `config.yaml` 静态驱动**的常驻进程（`cmd/agent/main.go:4` 注释明确"运行时不接收业务命令行参数"；CLI 仅 `-config` / `-version`）。
- 启动即 `source.DiscoverCDRDailyTables` 发现**全部** `e_cdr_YYYYMMDD` 日表（`main.go:133` 调 `CDRTables(ctx)` 传 `nil`），每张起一个实时 `Puller`。
- 历史回填**无专门通道**：历史日表 = 从 flowno 0 起的普通增量 `Puller`（`puller.go` 的 `pollOnce` 按 flowno 水位增量 → 包 Kafka `Envelope` 推 `vos.cdr.live` → `state.json` 记 maxFlowno → 推平空闲、重启续传）。
- **死代码/未接线**：`config.Backfill.DateRange / Parallel` 在运行路径里未被引用（`main.go:133` 传 `nil`）；`state.MarkDone` 全代码零调用。
- 唯一入站面是可选 `server.listen` HTTP（测试拓扑，平台来拉），**无服务端→agent 下发指令/配置通道**。

### 1.2 目标（用户需求）

1. agent 本地要有**推送历史记录**，推送前先检查。
2. 默认启动**先推最新数据**（live-first）。
3. 后台进程**扫描历史数据量**，并**推送消息给服务端**告知"有多少历史可推"。
4. 服务端让用户**选择是否接收** + **接收方式**（立即 / 定时 cron）。
5. **一次不推太多**、**不影响 VOS 正常运行**，建议用户在**非业务高峰期**操作。
6. 推送完成在**本地历史标记**，避免重复推送。
7. **实时数据必须始终正常推送**（最高优先级），历史数据的取舍完全交给服务端。
8. agent 要**支持接收服务端控制指令**：暂停 / 启动(恢复) / 重新扫描 等（作用于历史回填与扫描，不影响 live）。

### 1.3 设计原则

- **agent 只出站**：所有到服务端的通信都是 agent 主动连 `:3001`（Kafka）。不新增任何 inbound 端口。
- **零压 VOS**：扫描用 `information_schema.tables.TABLE_ROWS` 估算，不 `COUNT(*)`；精确数仅按需单表限流执行。
- **幂等兜底**：即便重发，ClickHouse `ReplacingMergeTree` 按 flowno 去重。
- **服务端定夺（用户确认）**：agent 只负责「轻量扫描 + 推送可用性消息」；**同步哪些历史表、何时同步（立即 / 定时）完全由服务端决定并下发 `vos.agent.command`**。agent 不自行决定回填范围与时间，收到 command 才执行。
- **live 最高优先级（硬红线）**：实时推送（rolling `e_cdr` + 今日日表）永远正常运行，**任何控制指令（暂停/恢复/重扫）都只作用于历史回填与扫描，绝不触碰 live**。见 §3.4.1。
- **可控（双向指令）**：agent 需支持服务端下发的控制指令集（`backfill_start/pause/resume/cancel`、`rescan`、`precise_count`、`set_throttle`），并回 `ack` 确认。见 §3.4.2。

## 2. 总体架构

```
[VOS 本机 · ykvos-agent]                 [服务端 · yudao 中台]
 ⛔ live 永不受控 ─────────────────────────────────────────────┐
 ① 启动即推最新(live)  ═══════ 始终推送(最高优先级) ═══════▶ ClickHouse
    rolling e_cdr + 今日日表 → Kafka:3001(vos.cdr.live)         │
                                                                 │
 ② 后台扫描历史(轻量)        ──report──▶     存 availability     │
    TABLE_ROWS 估算，零压力    (avail/progress/ack)  UI：选表 + 立即/定时(cron)
                              ◀─command──   服务端决定：同步哪些表 / 何时同步
 ③ 受控回填 worker (可暂停)     (指令集)     + 控制按钮：暂停/恢复/重扫/调速
    max_batch 200 · 1 表并发    backfill_start/pause/resume/cancel
    窗口 02:00-06:00 · 令牌桶    rescan / precise_count / set_throttle
    完成 → push_history 标 done  → 建任务表 + 消费 ack/progress 更新 UI
```

- **控制面**：Kafka topic `vos.agent.report`（agent→server，含 availability + progress + ack）与 `vos.agent.command`（server→agent 的**指令集**，按 `vos_id` 分区）。**live 数据面与控制面完全隔离，控制指令不触碰 live。**
- **数据面**：不变 —— agent → Kafka:3001 → ClickHouse Kafka 引擎 → `vos_cdr_ods`。
- **对外端口**：仅 `3000`（用户 + 管理后台）/ `3001`（Kafka），**不新增**。

## 3. 模块设计

### 3.1 本地推送历史 `push_history.json`（新增，独立于 `state.json`）

- 与现有 `state.json`（flowno 水位，管断点续传）**分开**，降低回归风险。
- 结构：每历史日表一行
  ```json
  {
    "<e_cdr_YYYYMMDD>": {
      "first_flowno": 0,
      "last_pushed_flowno": 123456,
      "total_pushed": 123456,
      "status": "pending|approved|syncing|done|rejected",
      "mode": "immediate|scheduled",
      "scheduled_cron": "0 2 * * *",
      "reported_at": "2026-07-20T11:00:00+08:00"
    }
  }
  ```
- **推送前检查**：读取 `push_history`，`done` / `rejected` 直接跳过；`approved` 按水位增量追；避免重复。
- 落盘沿用 `state.go` 的原子写（临时文件 + rename）模式。

### 3.2 启动分层（live-first）

- 启动**只立即起** `rolling e_cdr` + **今日** `e_cdr_YYYYMMDD` 的实时 `Puller`（最新数据马上到）。
- 更早的历史日表进入**待回填清单**，**不立即推**，等扫描 + 服务端审批后由回填 worker 执行。
- 实现：`main.go` 的表发现拆为"live 表集合"与"历史日表集合"，历史集合暂不起 worker。

### 3.3 轻量扫描 + 精确数（agent 侧，仅"扫 + 报"，不决策）

- **职责边界（确认）**：本步骤 agent **只做轻量扫描并把结果推成消息给服务端**；**"同步哪些表、何时同步"是服务端的事**，agent 拿到 `vos.agent.command` 后才执行。
- **后台 goroutine**（live 跑起来后执行一次；并周期性重扫，默认每 24h，捕获新出现的日表）：
  - `SELECT TABLE_ROWS FROM information_schema.tables WHERE table_name LIKE 'e_cdr_%'` **估算**每个历史日表行数（不 `COUNT(*)`，对 VOS MySQL 零压力）。
  - 结合本地 `push_history` 算"尚未推送/待批准"的可用量。
  - 组装 availability 报告，生产到 `vos.agent.report`（`msg_type=availability`）。
- **精确数（按需）**：用户在前端点开某日表详情时，服务端下发 `precise-count` command → agent 对该表执行**一次性 `COUNT(*)`**，但**限流 + 强制非高峰窗口**执行，结果经 `vos.agent.report`（`msg_type=progress`）回填展示为"估算 X / 精确 Y"。

### 3.4 Kafka 控制通道（替代 HTTP）

| Topic | 方向 | 内容 | 说明 |
|---|---|---|---|
| `vos.agent.report` | agent → server | availability / progress / ack | `msg_type` 区分；server 消费落库 |
| `vos.agent.command` | server → agent | **指令集**（回填任务 + 控制指令） | **`vos_id` 做消息 key/分区**，每 agent 只取自己实例的指令 |

- **鉴权**：topic 走现有 Kafka SASL（复用 `kafka.sasl` 段；yudao 侧配同样 SASL）。
- **仍 agent 只出站**：agent 生产 report、消费 command，均连 `:3001`，无 inbound。
- **消费组隔离（新增优化）**：`command_group` 命名必须硬包含 `vos_id`（例如 `ykvos-agent-group-vos1`），以确保多节点部署时，各 Agent 只订阅和消费本实例的指令，避免控制指令在节点间轮询消费导致指令丢失。

#### 3.4.1 ⛔ 硬红线：live 实时推送永不受控制指令影响

- **实时数据（rolling `e_cdr` + 今日日表）的推送是 agent 的最高优先级职责，任何控制指令都不能暂停/降速它。**
- `pause` / `resume` / `rescan` 等控制指令的作用域**仅限历史回填 worker 与扫描器**，与 live Puller 完全隔离（不同 goroutine、独立开关）。
- 唯一例外：进程整体 `stop`（运维停机）会同时停 live，但那是关进程、非"控制指令"，且需明确高危确认。

#### 3.4.2 command 指令集（server → agent）

每条 command 用 `action` 字段区分。agent 消费后按 `action` 分派，并回 `msg_type=ack` 确认（含 `command_id` + 执行结果）。

| `action` | 作用域 | 语义 |
|---|---|---|
| `backfill_start` | 历史回填 | 下发一个审批后的回填任务（含 tables[]、mode=immediate/scheduled、cron），排入回填队列 |
| `backfill_pause` | 历史回填 | **暂停**当前/全部回填 worker（live 不受影响）；已推批次保留水位，可续 |
| `backfill_resume` | 历史回填 | **恢复**被暂停的回填，从 `push_history` 水位续推 |
| `backfill_cancel` | 历史回填 | 取消指定 `task_id`；该表状态回 `pending`（未来可重新下发） |
| `rescan` | 扫描器 | **立即重新扫描**历史日表（不等 24h 周期），生成最新 availability 报告；捕获新增日表/行数变化 |
| `precise_count` | 扫描器 | 对指定日表按需 `COUNT(*)` 精确计数（限流 + 非高峰），结果回 progress |
| `set_throttle` | 历史回填 | 运行时调节流档（max_batch / rows_per_minute / active_window 等），无需改 config 重启 |

- **幂等**：每条 command 带唯一 `command_id`；agent 本地记已执行的 `command_id`，重复投递只回 ack 不重复执行。
- **暂停/恢复的落地**：回填 worker 用一个可中断的控制开关（`context.Context` + pause channel）；`backfill_pause` 置暂停位、worker 在**批次边界**（每 200 行一批的间隙）检查并挂起——保证暂停不会撕裂正在推的批次，水位一致。

- **消息 schema 示例**：
  ```json
  // report: 可用性
  {"vos_id":"vos1","msg_type":"availability",
   "tables":[{"table":"e_cdr_20260101","estimated_rows":88213,"already_pushed":0}],
   "generated_at":"2026-07-20T11:00:00+08:00"}
  // report: 进度
  {"vos_id":"vos1","msg_type":"progress",
   "task_id":"t-123","table":"e_cdr_20260101","pushed":5000,"status":"syncing"}
  // report: 指令确认
  {"vos_id":"vos1","msg_type":"ack",
   "command_id":"c-789","action":"backfill_pause","result":"ok","at":"2026-07-20T11:05:00+08:00"}

  // command: 下发回填任务
  {"vos_id":"vos1","command_id":"c-456","action":"backfill_start",
   "tables":["e_cdr_20260101","e_cdr_20260102"],"mode":"scheduled","cron":"0 2 * * *"}
  // command: 暂停回填（live 不受影响）
  {"vos_id":"vos1","command_id":"c-789","action":"backfill_pause"}
  // command: 立即重新扫描
  {"vos_id":"vos1","command_id":"c-790","action":"rescan"}
  
  // report: 心跳与健康状态上报 (msg_type=heartbeat)
  {
    "vos_id": "vos1",
    "msg_type": "heartbeat",
    "generated_at": "2026-07-20T11:50:00+08:00",
    "agent_version": "1.0.4",
    "vos_system": {
      "hostname": "vos-server-01",
      "os": "CentOS Stream 8",
      "cpu_load_1m": 0.85,
      "cpu_cores": 8,
      "mem_total_mb": 16384,
      "mem_used_mb": 8192,
      "disk_total_mb": 102400,
      "disk_used_mb": 40960,
      "uptime_seconds": 1234567
    },
    "db_status": {
      "connected": true,
      "version": "8.0.20",
      "open_connections": 3,
      "active_connections": 1
    },
    "agent_status": {
      "goroutines": 15,
      "mem_alloc_mb": 25.4,
      "uptime_seconds": 3600
    }
  }
  ```

### 3.5 服务端（yudao）侧（新增 `yk-vos-agent` 模块）

- **消费** `vos.agent.report`：
  - `msg_type=availability` → 落库 `vos_agent_backfill`（每实例每历史日表的 availableRows / lastReportedAt / status）。
  - `msg_type=heartbeat` ➔ 写入 VOS 实例状态缓存 / 实例状态表，更新监控大屏展现的 VOS 主机物理健康度、数据库存活与延迟（取代原有的 `/healthz` HTTP 直接探测，极度适用于无入站公网的测试环境安全监控）。
- **管理后台新页「历史话单回填」**（web-ele）：
  - 列各 VOS 实例 + 可回填日表 + 行数（估算 / 精确）+ 状态。
  - 操作：**立即接收** / **定时接收(cron)** 按钮（选表 + 定时间，下发 `backfill_start`）。
  - **控制按钮**：暂停(`backfill_pause`) / 恢复(`backfill_resume`) / 取消(`backfill_cancel`) / **重新扫描**(`rescan`) / 调速(`set_throttle`)。
- **生产** `vos.agent.command`：用户确认后建任务/发指令（`vos_agent_backfill_task` 表，含 mode / cron / status；每条指令带 `command_id`）。
- **收 progress / ack**：agent 回报经 `vos.agent.report` 更新任务状态、指令执行结果与 UI。
- 属"把定制功能迁进 yk-vos-new"范畴，工作量较大（后端 + 前端新模块）。

### 3.6 受控回填 worker（节流、不压 VOS）

- agent 消费到 `vos.agent.command` 中本实例的 approved 任务后，对目标日表起**受控 `Puller`**：
  - 复用现有 `puller.Puller.drain / pollOnce`（flowno 增量 + `state.json` 水位 + 幂等），仅替换参数为保守档。
  - **保守档（已定）**：
    | 参数 | 值 |
    |---|---|
    | `max_batch_size` | 200 |
    | `max_concurrent_tables` | 1（一次只回填一张日表） |
    | `max_rows_per_minute` | ≈ 10000（令牌桶限速） |
    | `idle_between_batches_ms` | ≈ 500 |
    | `active_window` | 02:00–06:00 |
    | 回填 MySQL 连接数 | 降到 2（与 live 错峰，全局 `max_open_conns=5` 不动；回填走独立低优先级连接） |
  - 所有参数在 `config.yaml` 可改，且可被 `set_throttle` 指令运行时热调。
- **窗口门控**：scheduled 任务严格卡 `active_window`；immediate 任务在下个窗口开始执行（见 3.7）。
- **可暂停/可恢复（对接 3.4.2 指令）**：回填 worker 由一个 `context.Context` + 暂停开关驱动，在**每批（200 行）边界**检查暂停位——
  - `backfill_pause` → 置暂停位，当前批推完即挂起（不撕裂批次，`push_history` 水位保持一致），回 `ack`。
  - `backfill_resume` → 清暂停位，从 `push_history.last_pushed_flowno` 续推。
  - `backfill_cancel` → 取消 context，worker 退出，该表状态回 `pending`。
  - **live Puller 与回填 worker 是独立 goroutine + 独立开关**，暂停回填绝不影响 live。
- **批次边界安全控制（新增优化）**：`backfill_pause` 指令生效时，回填 Worker 必须在**当前批次的数据完全发送成功、ClickHouse 返回写入确认后**，才更新 `push_history.json` 的最大流水号，然后安全挂起。这保证了高水位的绝对一致，杜绝了断点空隙和漏同步。
- **ClickHouse 合并优化（新增优化）**：在令牌桶限速回填时，为了避免高频小批次写入导致 ClickHouse 产生过多的 Parts（从而触发 `Too many parts` 报错），中台在向 ClickHouse 写入数据时，虽限速为每分钟 10,000 条，但数据发送模块应当在内存中积攒够 2,000 条（单次最大 limit）后再整体 POST 写入，确保“大批次、低频率”写入，减缓 CH 合并压力。

### 3.7 立即 / 定时 都卡非高峰

- **立即接收** = 系统在**下一个非高峰窗口开始**自动执行（用户免填 cron，一键即可），仍受速率/批次限制。
- **定时接收** = 用户指定 cron（UI 提示风险）。
- 两者皆窗口门控；区别仅在"是否要用户填时间"。

### 3.8 完成标记 + 去重兜底

- 日表追平（本批 < batch）后，`push_history` 标 `status=done` → 下次扫描不再报该表 → **不重复推送**。
- 即便重发，ClickHouse `ReplacingMergeTree` 按 flowno 去重兜底。

## 4. 端口与配置影响

- **不新增对外端口**：控制走 `3001`（Kafka，已有）；数据走 `3001`；用户/管理走 `3000`（已有）。
- `config.yaml` 新增/调整段（agent 侧）：
  ```yaml
  sync:
    backfill:
      enabled: true
      scan_interval_hours: 24      # 周期性重扫
      max_batch_size: 200
      max_concurrent_tables: 1
      max_rows_per_minute: 10000
      idle_between_batches_ms: 500
      active_window: "02:00-06:00"
      max_open_conns_backfill: 2
    heartbeat_interval_seconds: 30 # 新增：VOS 服务器与数据库状态心跳上报周期
  kafka:
    report_topic: "vos.agent.report"
    command_topic: "vos.agent.command"
    command_group: "ykvos-agent-vos1"   # 消费组，必须硬包含 vos_id 保证各 agent 独立消费本实例指令
  ```
- `state` 段新增：
  - `push_history_file`（默认与 `state.json` 同目录的 `push_history.json`）。
  - `command_log_file`（记已执行的 `command_id`，用于指令幂等去重）。

## 5. 工作量、版本与风险

- **agent 侧**：中等偏上（新增 scanner + `push_history` + Kafka **report 生产 / command 消费** + **指令分派器**(pause/resume/cancel/rescan/precise_count/set_throttle) + 可暂停回填控制器 + 保守档节流，约 500–800 行 Go）。可独立自测，版本升 **1.0.6**，重打服务端包分发。
- **yudao 后端 + 前端**：较大新模块（消费 report/ack、任务表、command 生产、管理后台页含选表+定时+控制按钮）。
- **风险**：
  - `TABLE_ROWS` 为估算值（InnoDB 抽样、MyISAM 精确）；精确数需 `COUNT(*)`，已限制为按需单表限流非高峰。
  - scheduled 依赖 agent 时钟 + Kafka 消费频率；任务错过窗口顺延至下一窗口。
  - Kafka 控制面引入 SASL 凭据管理（复用现有机制）。
  - **指令幂等**：Kafka at-least-once 会重投 command，必须靠 `command_id` + `command_log_file` 去重，否则 pause/resume 可能乱序或重复执行。
  - **live 隔离**：需在代码层严格保证控制指令只操作回填/扫描 goroutine，live Puller 独立开关——这是最高优先级不变量，实施与测试都要单独验证。
- **待办**：方案定稿后，由用户拍板从 agent 侧或全栈动手实施。
