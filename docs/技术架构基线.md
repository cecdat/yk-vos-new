| ClickHouse | ClickHouse | ODS 原始层 + DWS 汇总层，承载分析查询 | 分析只读 |

- VOS MySQL 是**只读源**，中台绝不回写业务数据（回写对接走 HTTP 适配器，见功能框架 Phase2）。
- 分析查询**只走 ClickHouse**，不让 VOS 库承担报表压力。
- 中台自有业务库与 VOS 库物理分离，互不影响。

## 2. 数据流向（端到端）

```
VOS MySQL (e_cdr + 51日表 + e_customer/gatewaymapping/feerate)
      │  ① 批量增量读取（BatchSource，Quartz 定时）
      │     - 按 manifest 遍历 51 个日表 + 滚动 e_cdr
      │     - 去重键 flowno；水位 = 已同步最大 flowno / starttime
      ▼
ClickHouse ODS 层（vos_cdr_ods 等，ReplacingMergeTree by flowno）
      │  ② 物化/汇总（DWS：vos_cdr_daily_customer 等）
      ▼
API 查询层（module-vos controller → service → ClickHouse JDBC / 业务库）
      │  ③ HTTP /admin-api/vos/**
      ▼
前端（views/vos/* → api/vos/* → @vben/request → 图表/表格）
```

**两条读取路径（明确区分）**
1. **维度/配置类**（客户、网关映射、费率、VOS 实例）：量小、需实时，直接读业务库或近实时同步到 ODS 维度表。
2. **话单类（海量）**：必须经 ClickHouse ODS/DWS，绝不直查 VOS 库。

## 5. 同步机制详设（核心）

### 5.1 同步架构：VOS 侧导出器 → Kafka → ClickHouse（引擎无关）

- **导出器（VOS 侧，polling，非 CDC）**：独立常驻进程，专用只读账号连本地 VOS MySQL，按每张表 `max(flowno)` 水位增量拉取，批量 publish 到 Kafka。
  - 为何轮询而非 Binlog：VOS 全表 MyISAM，无行级 binlog，Canal 类方案根本不可用（已核实）。
  - 水位持久化：导出器本地记录各表已同步 `flowno` 高水位（文件/小状态表），支持重启续传。
- **Kafka**：topic `vos.cdr`，按日或网关分区；承担缓冲、削峰、重放。
- **ClickHouse 消费**：通过 `Kafka` 表引擎或独立消费者写入 `vos_cdr_ods`（ReplacingMergeTree by flowno，天然去重）。
- **维度/配置/小表（e_customer / e_gatewaymapping / e_feerate / e_limit_e164 / e_report* / e_payhistory / e_alarm* … 共约 30+ 张，详见 `VOS同步Agent规格.md` §0.5）**：**改由 VOS 侧 agent 统一出口**（在 VOS 本机就近读 → Kafka 小 topic / 直写 CH），**平台侧不再直连 VOS 只读源**；仅当某实例网络隔离无法部署 agent 时，才退化为平台 `BatchSource` 应急。详见 agent 规格 §8。

### 5.2 回溯清单 `scripts/cdr_backfill_manifest.json`

- 覆盖 `e_cdr_20260118` ~ `e_cdr_20260309` 共 **51 个同构日表** + 滚动 `e_cdr`。
- 去重键 `flowno`；每个表记录 `watermark`（已同步最大 flowno）以支持断点续传。
- 51 个日表**结构完全相同**，SQL 用动态表名参数化，避免 51 份重复代码。

### 5.3 调度与职责划分

- **VOS 侧导出器**：常驻进程，自行按水位轮询并推 Kafka（不依赖平台 Quartz）。
- **平台侧（module-infra Quartz）**：仅负责「话单同步」页的同步状态/延迟监控（读 Kafka lag / CH 写入水位），**不再直连 VOS 拉维度**（维度改由 agent 统一出口，见 agent 规格 §8）；审计视图可选读 `e_user`/`e_syslog` 镜像。
- 同步日志落业务库（开始/结束/影响行数/失败表），前端「话单同步」页可读此日志。

## 6. 关键技术决策（ADR）

### ADR-001 · JDK17 与 Maven 隔离
- **背景**：本机 `JAVA_HOME` 默认 JDK11，而 `ruoyi-vue-pro@master-jdk17` 需 JDK17；全局 Maven 在本机 Git Bash 因 `MSYS_NO_PATHCONV=1` 无法启动。
- **决策**：项目专用 `tools/mvn`（cygpath 转 Windows 路径 + 写死 JDK17 Temurin 17.0.19）；永不调用全局 `mvn`。
- **后果**：+ 编译稳定可复现；− 团队成员需复用同一 `tools/` 包装，不能直接用系统 Maven。

### ADR-002 · 同步采用「VOS 侧轮询导出器 → Kafka」，放弃 Binlog/Canal（已核实 MyISAM）
- **背景（已核实）**：读取 `vos3000_structure.sql`，**全部表均为 `ENGINE=MyISAM`**。MyISAM 非事务、不产出行级 binlog，Canal / Debezium / Maxwell 均无法做增量 CDC；原设计文档的「Binlog + Kafka」前提不成立。（另：用户提供的同类产品 `capture` 插件——Go 编译的 VOS 侧轮询 agent、经 RabbitMQ 缓冲、ES 存储——**进一步证实该路线可行**，实现细节见第 10 节。）
- **决策**：在 VOS 服务器侧部署**轻量轮询导出器（polling exporter，独立进程/服务，非 MySQL 插件）**，用专用只读账号按 `flowno` 水位轮询 `e_cdr` + 51 日表，增量发布到 Kafka（topic `vos.cdr`）；ClickHouse 经 Kafka 引擎/消费者写入 ODS。保留 Kafka 作缓冲与重放。
- **后果**：+ 与存储引擎无关、可完整回溯、断点续传、Kafka 提供削峰/重放；− 需在 VOS 生产机上部署运维一个 agent（建议 **Go 静态二进制、零依赖**，Windows 下以服务运行），实时性为 T+秒/分钟级（足够运营分析）。
- **替代已否决**：Canal/Binlog 流式（前提不成立）；平台侧直连 BatchSource 拉全量（无缓冲、JDBC 跨网拉巨量行、无重放）。

### ADR-003 · ClickHouse 承载分析、MySQL 仅存业务
- **背景**：话单量极大，报表类查询若压在 VOS/业务库会拖垮生产。
- **决策**：ODS/DWS 全量落 ClickHouse；报表/统计/风控只查 CH；配置类小表可直查业务库。
- **后果**：+ 查询性能数量级提升；− 引入 CH 运维与双写一致性考量（用 ReplacingMergeTree by flowno 天然去重）。

### ADR-004 · 不复用 VOS 的 e_user，自建 RBAC
- **背景**：VOS `e_user` 是话务账号体系，与中台运营人员账号无关。
- **决策**：中台鉴权完全基于 ruoyi `module-system` 的 `sys_user/sys_role/sys_menu`，VOS 库只读。
- **后果**：+ 权限模型干净、可复用 ruoyi 代码生成器；− 需在中台单独维护运营账号。

### ADR-005 · 同步源可插拔
- **背景**：未来可能换数据源（如 VOS 升级 InnoDB 支持 CDC，或接入多个 VOS 实例）。
- **决策**：`CdrSource` 接口抽象，Batch/Canal 各自实现，Job 仅依赖接口。
- **后果**：+ 扩展成本低；− 接口设计需稳定，避免过度抽象。

## 10. 参考实现分析：同类产品 VOS 侧 capture 插件

> 来源：`tmp/capture-1.0.0.tar.gz` + `tmp/VOS数据同步程序部署说明.docx`（用户提供的同类产品部署包，仅作**设计参考**，不取其代码/凭据）。

### 10.1 它怎么实现的（已拆解）
- **形态**：VOS 服务器本机安装一个 **Go 编译的常驻 agent**（ELF 64-bit、`seelog` 日志库、init.d 服务 `service capture start/stop/restart`）。`install.sh` 把程序拷到 `/dipcc/bin/`、注册 init.d 并设开机启动，日志 `/var/log/capture.log`。
- **连 VOS 本机 MySQL**（配置段 `mysql`：`host/port/user/password/database/charset`）。关键：`database` 按版本填 `vos3000` / `vos5000`；并支持 `socket_path`——**部分 VOS 只能走 domain socket 访问 MySQL**（重要部署坑）。
- **轮询节奏**（配置段 `sync`）：`interval`（秒，默认 5）、`batch_size`（每次话单条数，文档默认 2000）、`start_date`（回溯起点，`YYYYMMDD`）、`local_ip`。→ 与我们的「flowno 水位 + 51 日表 manifest」**完全一致**。
- **消息缓冲**：用 **RabbitMQ**（不是 Kafka）。配置段 `rabbitmq` 含 `cdr`（exchange/queue/routing_key）与 `resource`（另一条流）两组 → **话单走 DB 轮询→RabbitMQ；资源/并发走 VOS API→另一条流**。
- **VOS API**（配置段 `vos`：`api_port` + `base_url`）：拉取 VOS 实时**并发数据**（不走 DB）。
- **存储端**：配置段 `ES`（Elasticsearch）—— 参考实现把数据最终落到 **Elasticsearch**（非 ClickHouse）。另有一段被注释的 `upload`（ASR 录音上传 + 转码路径），对应我们的 ASR 质检（Phase3）。
- **两条消息流**：`cdr`（话单，DB 轮询）/ `resource`（资源/并发，VOS API）—— 按数据来源分流，设计清晰。

### 10.2 可借鉴点（直接复用其设计）
1. **VOS 侧 Go agent 模式被证实可行** —— 与 ADR-002 完全一致，且是真实上线方案。
2. **5 秒轮询 + 批量 + start_date 断点** —— 我们的水位/manifest 可照搬，配置字段命名也可对齐。
3. **socket 兜底** —— 必须支持 `socket_path`，否则部分 VOS 连不上 MySQL。
4. **按来源分流**：DB 轮询（话单）与 VOS API（并发）分开两条流，互不阻塞。
5. **init.d/systemd 服务化 + 滚动日志** —— 运维范式可直接借鉴（Windows 侧则注册为服务）。
6. **ASR 上传段预留** —— 印证 Phase3 ASR 质检该走「录音拉取 + 转码 + 上传」链路。

### 10.3 与我们规划的差异 & 选型结论（已评估）

用户确认本机 VOS3000 为 **Linux**；消息中间件与分析存储经评估后**维持原规划（Kafka + ClickHouse）**。

| 维度 | 参考实现 | 我们决策 | 理由 |
|------|-----------|---------|------|
| 消息缓冲 | RabbitMQ | **Kafka**（维持） | 与 ClickHouse 有原生 Kafka 表引擎，ingestion 代码最少；retention/重放对计费话单的"丢一天不可接受"是刚需；已是原设计 |
| 分析存储 | Elasticsearch | **ClickHouse**（维持） | CDR 是聚合/报表密集负载，列存压缩 + 极速 SUM/COUNT/GROUP BY 远超 ES；ES 仅在对 ASR 转写文本做全文检索时后续可单独立 ES，不作主库 |
| 部署 OS | Linux（init.d） | **Linux（init.d/systemd）** | 已确认 VOS3000 为 Linux；agent 直接照搬参考的 init.d 服务形态 |
| 并发数据 | VOS API 拉取 | **采纳：VOS API** | 实时并发走 API，不进 DB 轮询，与话单流解耦 |
| 安全 | 明文密码 + root 直连 | **只读账号 + 密钥管理** | 见 10.4，绝不照搬明文 |

### 10.6 评估小结（为何维持 Kafka + ClickHouse）
- **两选型强耦合**：参考用 RabbitMQ+ES 是一套自洽栈；我们既已定 ClickHouse，则 Kafka 比 RabbitMQ 更契合——CH 能直接消费 Kafka（无需自写消费者），而 RabbitMQ 反而要额外写一层消费者落 CH。
- **体量不是分水岭**：CDR 速率 ~400/s 远未到临界点，二者都够；真正拉开差距的是 **CH 原生 Kafka 集成** 与 **计费话单的重放安全**，这两点都指向 Kafka。
- **参考的最大价值在 agent 形态而非中间件**：它证明了"VOS 侧 Go 轮询 agent + 5s 轮询 + socket 兜底 + DB/API 分流"可行；中间件与存储我们按自身栈（CH）择优即可。

### 10.4 安全改进（参考实现需修正之处）
- 参考 `config.yaml` 含**明文密码/ES 密码**且用 `root` 直连 VOS MySQL —— 我们**绝不照搬**：
  - VOS 侧只给**专用只读账号**（你已计划在 VOS 建独立用户），最小权限。
  - 消息中间件/RabbitMQ/ES 凭据走**密钥管理/环境变量**，不进仓库与镜像。
  - 连接串强制 `readOnly=true` + 网络白名单。

### 10.5 结论
参考实现**强有力验证了「VOS 侧轮询导出器」是正确路线**，且其 5s 轮询 / start_date 断点 / socket 兜底 / DB↔API 分流等细节可直接作为我们 `module-vos` 同步 agent 的设计蓝本。唯一需要你拍板的是**消息中间件（Kafka vs RabbitMQ）与分析存储（ClickHouse vs ES）** 两项选型，以及**本机 VOS 的操作系统**（决定 agent 以 Linux 服务还是 Windows 服务形态部署）。
