## 四、Agent 改造需求汇总

为实现以上功能，Agent（Go）需要以下改造：

| # | 改造点 | 当前状态 | 目标 |
|---|--------|---------|------|
| A1 | **多表同步** | 仅发现/同步 `e_cdr` 滚动表 | config.yaml 声明式表列表，每张表独立游标/批次/频率 |
| A2 | **维度表全量镜像** | 无 | 对 `e_customer`/`e_gatewaymapping`/`e_feerate` 以外的维度表做定期全量快照（Replace into CH） |
| A3 | **增量变更检测** | 无 | 对小表（`e_customer`, `e_alarm_*`）用 `lastupdatetime` 或 `MAX(id)` 做增量拉取 |
| A4 | **Schema 自动适配** | 手动写死 53 字段 | 从 VOS 信息_schema 或首行推断列类型，减少手动维护 |
| A5 | **同步状态上报** | 无 | 向平台汇报"某表上次同步时间/行数/延迟秒数"，供监控页展示 |

**建议优先级**：A1 > A2 > A3 > A5 > A4

## 五、前端技术选型确认

基于项目现状（RuoYi-Vue + Element Plus + Vue 3）：

- **框架**：保持 RuoYi-Vue 前端脚手架不变
- **UI 库**：Element Plus（与截图风格一致）
- **图表**：ECharts（趋势图/饼图/热力图）
- **表格大数据**：`el-table-v2`（虚拟滚动，话单 10 万行不卡）
- **路由**：沿用 RuoYi 动态路由 + 菜单配置
- **状态管理**：Pinia

## 六、风险与注意事项

1. **VOS MySQL MyISAM 锁**：大表查询会阻塞写入，agent 批量 SELECT 需控制并发（当前 max_open_conns=5 是对的）
2. **e_cdr 滚动表分区**：VOS 按日建表 `e_cdr_YYYYMMDD`，agent discovery 需持续发现新表
3. **实时性折衷**：财务管理需要近实时的 `money`/`consumption`，考虑对关键字段做更高频轮询（如 30 秒），或通过 VOS Web API 推送
4. **ASR 数据量**：`e_aas_cdr.audiotext` 是 text 字段，CH 存储成本高，建议仅保留最近 N 天或存对象存储
5. **号码隐私**：前端展示号码需脱敏处理（中间 4 位掩码），导出时可选明文
