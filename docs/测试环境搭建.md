# YK-VOS 测试环境搭建

## 0. 拓扑与约束（先读）

| 角色 | 位置 | 公网 | 部署内容 |
|---|---|---|---|
| **VOS 服务器（可多台）** | 客户侧/测试 VOS | **有公网 IP** | 每台一个 `ykvos-agent`（`sync.mode=off` + `server.listen=:5233`） |
| **平台测试服务器** | 我们部署 | **无公网入站端口** | ClickHouse / Kafka / **监控页 monitor** / (Redis / Postgres / 后端 预留) |

**关键约束**：平台服务器**没有公网入站端口**，所以：
- 生产里「agent 出站推 Kafka」在此**不可行**（Kafka 在平台侧，agent 连不进来）。
- 改为**平台主动出站去拉**：agent 在 VOS（有公网）起 HTTP 拉取端点，平台侧的 `pull_client.py` / 监控页 出站 GET 它，拉回的话单落**本机 ClickHouse**。
- 这样既满足「VOS 有公网 / 平台无公网端口」的现实，又**不暴露任何入站端口到平台**（连接方向是平台→VOS，符合 §9 安全基线）。

**多节点**：每个 VOS 实例各跑一个 agent，靠 `instance.id`（即 Envelope 的 `vos_id`）区分。
`flowno` 在单台 VOS 内唯一、跨 VOS **不唯一** → ClickHouse `vos_cdr_ods` 去重键为 **(vos_id, flowno)**，监控页按 `vos_id` 分节点展示链路 lag。

## 1. VOS 侧：装 agent（1.0.3+，每台一份）

```bash
# 在每台 VOS 服务器执行（每台 instance.id 必须唯一：vos1 / vos2 ...）
tar -xzf ykvos-agent-1.0.3.tar.gz && cd ykvos-agent-1.0.3
sudo bash install.sh

# 测试 VOS 查询账号：user=vosdev，密码经 YK_MYSQL_PWD 注入（§9 密码不进 config）
export YK_MYSQL_PWD='21Hnykxx@2026'     # 测试环境凭据；生产请改用专用只读账号并轮换

# 改 /opt/ykvos-agent/etc/config.yaml
#   instance.id / name   -> 本台唯一标识，须与监控页 NODES 的 name 一致（如 vos1）
#   mysql.user            -> vosdev（测试）；生产替换为 yk_vos_ro 类只读账号
#   sync.mode: "off"          # 不推 Kafka，仅保留 serve 端点（平台来拉）
#   server.listen: ":5233"   # 起 HTTP 拉取端点（VOS 有公网，平台可连）
#   server.token: ""           # 可选：设 YK_SERVER_TOKEN 做 Bearer 鉴权
# VOS 防火墙放行 5233（仅对平台服务器 IP 放开）
sudo /opt/ykvos-agent/bin/ykvos-agent -config /opt/ykvos-agent/etc/config.yaml
```

验证端点活：`curl http://127.0.0.1:5233/healthz` 与 `curl http://127.0.0.1:5233/v1/watermark?table=e_cdr`

## 2. 平台测试服务器：起数据底座

镜像（用户指定 `docker.1ms.run` 源）：
- `docker.1ms.run/clickhouse/clickhouse-server:latest`
- `docker.1ms.run/bitnami/kafka:3.4.1`
- `docker.1ms.run/redis:7`（后端用，默认注释）
- `docker.1ms.run/postgres:15`（后端用，默认注释）
- `pinny3/openjdk21:latest`（后端 jar 运行镜像，默认注释）

```bash
cd yk-vos-new
docker compose -f docker-compose.test.yaml config        # 先校验
docker compose -f docker-compose.test.yaml up -d clickhouse kafka monitor
```

`monitor` 随 compose 一起起（无需额外镜像，FROM scratch 跑预编译静态二进制）。
监控页读环境变量 `NODES`（节点列表）、`CLICKHOUSE_URL`（默认 `http://clickhouse:8123`）。

## 3. 建 ClickHouse 表（多节点版）

`vos_cdr_ods` 已加 `vos_id` 列、去重键 `(vos_id, flowno)`、按月分区；维度表 `vos_customer_ods` 等同样加 `vos_id`。

## 4. 平台侧：拉取并落库（两种 sink，支持多节点）

脚本：`agent/tools/pull_client.py`（零依赖，仅需标准库）。`--agent-url` 可重复多次（每个 VOS 节点一个）；每节点独立游标。

### 4.1 直写 ClickHouse（默认，最快验证）
```bash
python3 agent/tools/pull_client.py \
  --agent-url http://<VOS1公网IP>:5233 \
  --agent-url http://<VOS2公网IP>:5233 \
  --clickhouse http://127.0.0.1:8123
# 先 --once 拉一批看输出；确认无误后去掉 --once 持续拉。
```
- 行为：把 Envelope.data（列名→值的 JSON 对象）合并顶层 `vos_id` 后，以 `JSONEachRow` 直接 INSERT 进 `vos_cdr_ods`，**按列名映射**，复用生产 ODS 契约。
- 校验（按节点）：`SELECT vos_id, count() FROM vos_cdr_ods GROUP BY vos_id`

### 4.2 经本机 Kafka 中继（验证生产消费链路）
```bash
pip install kafka-python
python3 agent/tools/pull_client.py \
  --agent-url http://<VOS1公网IP>:5233 \
  --sink kafka --kafka-bootstrap 127.0.0.1:9099
```
- 行为：把整条 Envelope 发到本机 Kafka `vos.cdr.live` → ClickHouse 原生 Kafka 引擎(`vos_cdr_kafka`)消费 → MV(`vos_cdr_mv`)解析 `data` JSON → `vos_cdr_ods`。
- 与生产**完全一致**的消费者侧，仅生产者从「agent 直推」换成「平台中继」。

> 两种方式最终都汇入同一张 `vos_cdr_ods`，靠 `(vos_id, flowno)` 去重（`ReplacingMergeTree`），可任选其一；4.2 更贴近生产但多一个 Kafka 依赖。

## 5. 监控页：查看数据链路是否正常

`monitor` 服务（docker 形式运行）周期性探测：
- 各 VOS 节点 agent 的 `/healthz`（在线/离线）与 `/v1/watermark`（agent 侧当前 MAX(flowno)）；
- 本机 ClickHouse 的 `vos_cdr_ods` 总/分节点行数、MAX(flowno)、最后同步时间；
- **链路 lag = agent 侧水位 − ClickHouse 侧水位**，>0 表示落后，按节点展示。

查看（因平台服务器无公网端口，用 SSH 隧道转发后浏览器打开）：
```bash
# 在本地笔记本执行，把平台服务器的 8088 转到本地
ssh -L 8088:127.0.0.1:8088 user@平台服务器
# 然后浏览器开 http://127.0.0.1:8088
```
页面每 15 秒自动刷新。整体/各节点状态、CH 可达性、分节点 lag 一目了然。

`NODES` 配置示例（compose 中，name 须与 agent 的 `instance.id` 一致）：
```yaml
environment:
  NODES: "vos1=http://<VOS1公网IP>:5233,vos2=http://<VOS2公网IP>:5233"
  CLICKHOUSE_URL: "http://clickhouse:8123"
  CH_TABLE: "vos_cdr_ods"
  MONITOR_TOKEN: ""          # 若 agent 启用 Bearer 鉴权，填同值
```

## 6. 后续（后端启用时）
取消 `docker-compose.test.yaml` 中 `redis` / `postgres` / `backend` 注释，构建 `yudao-server.jar` 后挂载启动。后端经 **ClickHouse 原生端口 9000** 读 ODS/DWS，经 **Postgres 5432** 存业务数据，经 **Redis 6379** 缓存。

## 7. 注意
- ClickHouse 默认用户 `default` 无密码 —— **仅测试**；生产务必设强密码 + 网络隔离。
- Kafka `auto_create_topics_enable=true` 便于测试；生产建议显式建 topic 并启用 SASL/TLS。
- 平台服务器无公网端口，故 ClickHouse/Kafka/monitor 仅本机/容器网络可达（monitor 对 VOS 是出站），外部不可入站，符合 §9 安全基线。
