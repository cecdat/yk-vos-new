## 踩坑：含 `$` 的 BCrypt 密码哈希不能用 `bash -c` 双引号传（会损坏）
- **事件**：用户按我给的 `docker compose exec mysql bash -c " ... SET password='\$2a\$04\$...' WHERE ..."` 重置 admin 密码，执行「成功」但登录仍「密码错误」。
- **根因**：命令外层 `bash -c "..."`，内层 `-e "..."` 仍是**双引号**，bash 在双引号里展开 `$2`/`$0`/`$04`（当位置参数），把 BCrypt 哈希里的 `$2a$04$.vd...` 截改成损坏串写进库 → 密码永远不匹配。
- **正确写法（永不再用双引号套 `$`）**：用**引号 heredoc** 禁止任何展开，直接喂 mysql：
  ```bash
  docker compose exec -T mysql mysql -uroot -p123456 ykvos <<'SQL'
  UPDATE system_users SET password='$2a$04$.vd8nPeLwxt6hnSzmAoAyul8BOLX7Cib6QhcxRe30rfvrIPQHH1OG' WHERE username='admin';
  SQL
  ```
  单引号 `<<'SQL'` 是禁用展开的关键；`-T` 关 TTY 让 stdin 干净传入。验证：`SELECT username,password FROM system_users WHERE username='admin';` 应一字不差等于上面哈希。
- **附带提醒**：用户「完整初始化」后仍报 `infra_api_access_log` 不存在 → 当时 `ykvos` 库是空的（`system_users` 都不存在）。重置/登录前要先用 for 循环把 5 个 SQL 导入（顺序 quartz→ykvos→vos→vos_menu→vos_role_seed），否则一切 SQL 操作都会 `table doesn't exist`。导入后 admin 默认即 `admin123`。
- **第四次确认（用户贴 SELECT 0 行）**：用户用正确的 `<<'SQL'` heredoc 跑了 UPDATE + `SELECT username,password FROM system_users WHERE username='admin'`，**SELECT 返回 0 行（无 admin 记录）**。证实 `ykvos` 库**种子数据从未导入**——`UPDATE` 只能改已存在的行，没行就 0 row affected，所以密码永远错。这等于再次确认「空库/缺种子」根因未解决。唯一可靠解法仍是**清卷全量重导**：`docker compose down && rm -rf ./data/mysql ./data/clickhouse ./data/kafka && docker compose up -d`（必须带 `rm -rf`，否则 down 不删 bind mount、initdb.d 仍被跳过）。起来后 admin/admin123。本次用户此前大概率只在 `down/up` 没真正 `rm -rf`，故反复空库。
