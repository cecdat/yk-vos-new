# YK-VOS 整体部署手册（服务端 + Agent）

> 版本：服务端 `ykvos-server-dev-1.0.6` ｜ Agent `ykvos-agent-1.0.6`
> 适用范围：测试环境（dev）合并编排；生产（prod）同理，仅镜像/密码/端口策略不同。
> 本手册覆盖：**初始化安装、更新安装、安装顺序、注意事项**。

---

## 0. 系统组成与拓扑

本交付物是**自包含合并包**，同时包含「数据底座」和「应用层（前后端）」，另外每台 VOS 节点单独装 agent。

| 角色 | 部署位置 | 组件 | 说明 |
|---|---|---|---|
| **服务端（平台/测试服务器）** | 一台有 docker 的机器 | docker-compose 全栈 | MySQL/Redis/backend/frontend + ClickHouse/Kafka/monitor/init |
| **Agent 端** | 每台 VOS 服务器本机 | `ykvos-agent` 二进制 + init.d 服务 + 看门狗 | 就近读 VOS 本机 MySQL，把话单同步到服务端 |

### 拓扑（测试环境）
- VOS 服务器**有公网**；平台测试服务器**无公网端口**。
- 测试拓扑采用**双向/拉取模式**：每台 VOS 本机 agent 起 HTTP 拉取端点 `:5233`，平台侧 `tools/pull_client.py` 出站周期去拉，落本机 ClickHouse。
- 生产拓扑采用**推送模式**：agent 出站直推服务端 Kafka `:3001`，由 ClickHouse Kafka 引擎消费。

### 端口规划（服务端最小化暴露）
| 端口 | 服务 | 面向 | 说明 |
|---|---|---|---|
| **3000** | frontend(nginx) | 用户浏览器 | 唯一人机入口；反代 `/admin-api` → `backend:48080` |
| **3001** | kafka(EXTERNAL) | VOS 节点 agent | 【推送模式】agent 推话单入口；纯拉取模式可不放行 |
| **5233** | VOS 本机 agent | 平台 pull_client | 【拉取模式】在 VOS 侧放行，不在服务端 |
| 48080/3306/6379/8123/9000/8088 | 各内部服务 | 仅容器内 | **均不对外**，仅容器内用服务名互通；调试时临时加 `ports` |

### 关键账号与默认值
| 项 | 值 |
|---|---|
| 管理后台地址 | `http://<服务端IP>:3000` |
| 默认管理员 | `admin` / `admin123`（yudao 内置，首次登录后请改密） |
| MySQL | 库 `ykvos` ｜ 用户 `root` ｜ 密码 `123456` |
| Redis | `redis:6379` ｜ database `1` |
| 后端 API | 容器内 `backend:48080/admin-api`（前端同源反代，无需对外） |
| ClickHouse | `clickhouse:8123`（HTTP）/ `:9000`；库 `ykvos` |
| Kafka | 内部 `kafka:9092` ｜ 外部 `:3001` ｜ topic `vos.cdr.live` |
| Agent（测试） | 账号 `vosdev` ｜ 密码经 `YK_MYSQL_PWD` 注入（测试 `21Hnykxx@2026`） |

## 1. 环境要求（前置）

**服务端机器**
- Linux + Docker Engine ≥ 20.10 + docker compose v2（`docker compose` 子命令）。
- 磁盘：数据全部落在 `release/dev/data/`（MySQL/ClickHouse/Kafka/日志），首次建库约需数 GB，建议 ≥50GB 空闲。
- 镜像源（用户指定）：多数用 `docker.1ms.run/...`；后端 `pinny3/openjdk21:latest`（Docker Hub）；前端 `docker.1ms.run/nginx:latest`。若某镜像拉取失败，去掉 `docker.1ms.run/` 前缀改用官方源。
- 防火墙/安全组：放行入站 **3000**（用户）、**3001**（VOS，仅推送模式）。

**每台 VOS 节点机器**
- Linux（x86_64），无需预装运行时（agent 为 Go 静态二进制）。
- 本机 VOS MySQL 可达（`127.0.0.1:3306`，或 socket），且存在只读查询账号（测试 `vosdev`）。
- 防火墙：拉取模式下放行本机 **5233** 给平台侧 IP。

## 2. 安装顺序（总览）

> 原则：**先服务端、后 agent**；**先起底座、再起应用**；**先验证服务端自洽、再联调 agent**。

```
① 准备服务端机器（装 docker、确认镜像可拉）
② 准备各 VOS 节点（确认 VOS MySQL 只读账号、防火墙）
③ 服务端初始化：解包 → 配 .env → docker compose up（数据底座+应用层自动初始化）
④ 验证服务端自洽（容器 healthy、前端可开、admin 可登录）
⑤ 如库已存在需补建 VOS 数据：docker compose exec mysql bash /init/init-mysql-vos.sh
⑥ 每台 VOS 节点：装 agent → 配 config.yaml + .env 密码 → 放行防火墙 → 启动
⑦ 验证数据闭环（pull_client 拉取 或 Kafka 推送 → ClickHouse 有数）
```

## 3. 初始化安装

### 3.1 服务端（平台/测试服务器）

```bash
# 1) 取包并解压（得到 release/dev/ 目录结构）
tar -xzf ykvos-server-dev-1.0.6.tar.gz
cd release/dev

# 2) 生成运行配置
cp .env.example .env
vi .env
#   - 推送模式必填：YK_KAFKA_ADVERTISED_HOST=<VOS 能连到的服务端IP>  （默认 120.226.208.2）
#   - 拉取/双向模式填：NODES=vos1=http://<VOS1公网IP>:5233,vos2=http://<VOS2公网IP>:5233
#   - 其余（CLICKHOUSE_URL / CH_TABLE / MONITOR_TOKEN / PORT）保持默认即可

# 3) 起全栈（数据底座自动初始化，应用层等依赖 healthy 后启动）
docker compose up -d

# 4) 观察启动与初始化日志
docker compose ps
docker compose logs -f mysql backend        # 看建库/建表是否成功
```

**自动初始化机制（仅首次）：**
- MySQL 数据目录为空时，`backend/sql/mysql/` 挂载进 `/docker-entrypoint-initdb.d`，**首次启动自动按顺序执行**全部 `*.sql`：
  - `ykvos.sql`（yudao 基座建库）+ `quartz.sql`
  - `vos.sql`（VOS 4 张表，`CREATE TABLE IF NOT EXISTS`）
  - `vos_menu.sql` / `vos_role_seed.sql`（菜单/角色/套餐种子，`INSERT IGNORE` 幂等）
- ClickHouse：`clickhouse-init` 容器健康后自动跑 ODS 建表 + Kafka MV。
- Kafka：`kafka-init` 自动建 topic `vos.cdr.live`。
- 数据全部落在 `release/dev/data/`，容器删除数据仍在。

**验证服务端自洽：**
- `docker ps`：`yk-mysql`/`yk-redis` healthy、`yk-backend`/`yk-frontend` up、`yk-clickhouse`/`yk-kafka`/`yk-monitor` up；各 `*-init` 为 `restart:no`、跑完 Exit 0。
- 浏览器开 `http://<服务端IP>:3000` → 用 `admin/admin123` 登录成功。
- （可选）确认 VOS 表已建：`docker compose exec mysql mysql -uroot -p123456 ykvos -e "SHOW TABLES LIKE 'vos_%';"`

### 3.2 VOS 模块数据补建 / 重跑（库已存在时）

> ⚠️ 上面 MySQL 的自动机制**只在数据目录为空（首次 `docker compose up`）时跑一次**。
> 若库已存在（例如先装了标准 yudao、再补 VOS 模块），需手动补建：

```bash
cd release/dev
# 方式 A（推荐，免映射端口，在 compose 网络内执行）
docker compose exec mysql bash /init/init-mysql-vos.sh

# 方式 B（本机直连，需临时给 mysql 加 ports:["3306:3306"] 或 SSH 隧道）
MYSQL_HOST=127.0.0.1 MYSQL_PORT=3306 \
  MYSQL_USER=root MYSQL_PASSWORD=123456 \
  bash init/init-mysql-vos.sh
```
脚本仅执行 VOS 专属 3 个 SQL（`vos.sql`/`vos_menu.sql`/`vos_role_seed.sql`），**全部幂等，重复执行安全**（建表 `IF NOT EXISTS` + 种子 `INSERT IGNORE`）。

### 3.3 VOS 侧安装 Agent（每台节点）

```bash
# 1) 把 agent/ykvos-agent-1.0.6.tar.gz 拷到 VOS 服务器并解包
tar -xzf ykvos-agent-1.0.6.tar.gz && cd ykvos-agent

# 2) root 执行一键安装（拷贝运行时 → 注册 init.d 服务+看门狗 → 开机自启 → 启动）
sudo bash install.sh
#    安装位置：/opt/ykvos-agent/{bin,etc}
#    服务：/etc/init.d/ykvos-agent + /etc/init.d/ykvos-agent-monitor（看门狗，每30s拉起）
#    首次会因未配密码而连库失败，看门狗持续重试，写入密码后自动恢复，属正常。

# 3) 写入密码（密码不进 config.yaml，经 .env 注入；支持明文或密钥文件路径）
vi /opt/ykvos-agent/.env
#    YK_MYSQL_PWD='21Hnykxx@2026'          # 测试；生产务必换成专用只读账号密码
#    YK_SERVER_TOKEN='xxxx'                 # 拉取模式公网5233强烈建议设值（Bearer 鉴权）

# 4) 编辑配置（实例标识 / 同步模式 / Kafka 地址）
vi /opt/ykvos-agent/etc/config.yaml
#    instance.id: "vos1"                    # 每台 VOS 唯一，须与监控 NODES 的 name 一致
#    拉取/双向模式（测试）：
#      sync.mode: "off"
#      server.listen: ":5233"
#    推送模式（生产）：
#      sync.mode: "kafka"
#      server.listen: ""                     # 不起入站端点
#      kafka.brokers: ["<服务端IP>:3001"]   # 须与 YK_KAFKA_ADVERTISED_HOST 可达
#      kafka.sasl.mechanism: ""             # 留空=不启用

# 5) 重启使配置/密码生效
/etc/init.d/ykvos-agent restart

# 6) VOS 防火墙放行 5233（拉取模式）或确保能出网到服务端 3001（推送模式）

# 7) 查看状态与日志
/etc/init.d/ykvos-agent status
tail -f /var/log/ykvos-agent.log
```

> 说明：agent 进程退出后由 `ykvos-agent-monitor` 看门狗每 30s 自动拉起。若尚未写入密码，进程因连库失败退出，看门狗持续重试；写入密码后自动恢复。

## 4. 更新安装

### 4.1 服务端更新

发布包自带 `cleanup.sh`，更新前执行即可清理「跟随新包走的可重建文件」，**仅保留 `./data`（数据库/ClickHouse/Kafka/日志）与 `.env`（运行时配置）**。

```bash
cd release/dev
bash cleanup.sh            # 交互确认后删除；会先 docker compose down 释放挂载
# 或：bash cleanup.sh --force   （跳过确认）
# 或：bash cleanup.sh --dry-run （只列出将删内容，不实际删除）

# 清理后，把新的压缩包内容解压覆盖到本目录，再：
docker compose up -d
```

- 安全保护：脚本仅当目录内存在 `docker-compose.yaml` 才执行，防误删。
- 更新后 MySQL 数据目录已存在，自动初始化**不会重跑**；VOS 专属 SQL 幂等，可手动跑 `init-mysql-vos.sh` 补最新种子（菜单/角色变更时建议跑一次）。
- `.env` 不被新包覆盖，但新包可能新增环境变量；更新后对比 `.env.example` 补缴。

### 4.2 Agent 更新（每台 VOS 节点）

`install.sh` 设计为**保留现有配置**：已存在 `config.yaml` 会先备份（`config.yaml.bak.<时间戳>`）再保留，已存在 `.env` 直接保留，仅覆盖 `bin/ykvos-agent` 二进制。

```bash
# 重新解包新 agent 包，覆盖安装（配置不丢）
tar -xzf ykvos-agent-1.0.x.tar.gz && cd ykvos-agent
sudo bash install.sh
/etc/init.d/ykvos-agent restart
```
或最小化手动更新（不跑 install）：
```bash
# 仅替换二进制后重启
cp -f ykvos-agent/bin/ykvos-agent /opt/ykvos-agent/bin/ykvos-agent
chmod +x /opt/ykvos-agent/bin/ykvos-agent
/etc/init.d/ykvos-agent restart
```

## 5. 数据闭环验证

**拉取模式（测试）：**
```bash
# 平台侧直写 ClickHouse（零依赖，最快验证）
python3 tools/pull_client.py \
  --agent-url http://<VOS1公网IP>:5233 \
  --clickhouse http://127.0.0.1:18123 --once
# 验证落库：
docker exec -i yk-clickhouse clickhouse-client -m -q \
  "SELECT vos_id,count() FROM vos_cdr_ods GROUP BY vos_id"
```

**推送模式（生产）：**
- 确认 agent `config.yaml` 的 `kafka.brokers` 指向服务端 `:3001`，且服务端 `.env` 的 `YK_KAFKA_ADVERTISED_HOST` 为 agent 可达地址。
- 验证：服务端 `docker exec -i yk-kafka kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic vos.cdr.live --max-messages 1` 能看到消息；CH `vos_cdr_ods` 有数。

**监控页（链路健康 / lag）：**
测试服务器无公网端口，本机用 SSH 隧道：
```bash
ssh -L 18088:127.0.0.1:18088 user@<服务端IP>
# 浏览器开 http://127.0.0.1:18088
```
页面按 `vos_id` 分节点展示各 agent `/healthz`、`/v1/watermark`、CH 已落库行数 / 最后同步时间 / **lag = agent 水位 − CH 水位**。

## 6. 注意事项（务必阅读）

1. **首次 vs 已存在库**：MySQL 自动初始化只在数据目录为空时跑一次。库已存在再补 VOS 模块，必须手动 `init-mysql-vos.sh`。
2. **幂等安全**：所有 VOS SQL（`vos.sql`/`vos_menu.sql`/`vos_role_seed.sql`）均幂等，可重复执行；`cleanup.sh` 不会删 `./data` 与 `.env`。
3. **密码不落配置**：agent 的 MySQL/Kafka/Token 密码统一经 `/opt/ykvos-agent/.env` 注入（`YK_MYSQL_PWD` 等），**不要写进 config.yaml**。`.env` 权限 `600`。生产强烈建议用「密钥文件路径」形式（`YK_MYSQL_PWD=/path/to/secret`），天然免疫 `@/#/$` 等特殊字符。
4. **推送模式广播地址**：`.env` 的 `YK_KAFKA_ADVERTISED_HOST` 必须填「VOS 的 agent 能访问到的服务端 IP/域名」，填 `127.0.0.1` 会导致 agent 连不上、推不进来。
5. **实例标识唯一**：每台 VOS 的 `instance.id` 必须唯一，且须与监控 `NODES` 的 `name`、ClickHouse 去重键 `vos_id` 完全一致，否则数据串节点。
6. **防火墙最小暴露**：服务端仅放行 3000（用户）和 3001（VOS，推送模式）；5233 在 VOS 侧放行（拉取模式）。其余端口不对外。
7. **改密与账号**：默认管理员 `admin/admin123` 首次登录后请改密；测试用 `vosdev`/`21Hnykxx@2026` 仅为测试凭据，生产必须替换为独立只读账号并定期轮换。
8. **标准 yudao 不含定制页**：本合并包为**标准 yudao 后台**（登录/系统管理），**不含**在另一仓库 `yk-vos` 里做的「话单查询、VOS 管理页、云客vos 标题、去租户/其他登录方式」等定制。如需这些功能须先迁移再重新打包。
9. **数据持久化**：所有数据在 `release/dev/data/`，升级/重装务必保留；误删 `./data` 等同于清空数据库。
10. **镜像源回退**：某镜像 `docker.1ms.run/...` 拉取失败时，去掉前缀用官方源即可，不影响编排。
11. **agent 看门狗**：进程异常退出由 monitor 每 30s 拉起；若长期连不上库，先查 `.env` 密码与 VOS MySQL 账号权限，而非反复重启。
12. **日志位置**：服务端后端日志在 `release/dev/data/backend-logs/`；agent 日志在 `/var/log/ykvos-agent.log`。

## 7. 目录说明（服务端 release/dev/）

| 路径 | 作用 |
|---|---|
| `docker-compose.yaml` | 合并编排：数据底座(CH+Kafka+monitor+init) + 应用层(mysql+redis+backend+frontend) |

| `monitor/` | 监控页（Dockerfile + 二进制 + 前端源码） |
| `backend/` | `yudao-server.jar`(运行物) + `sql/mysql/*.sql`(建库) |
| `frontend/` | `dist/`(vben 静态站) + `nginx.conf`(反代配置) |
| `tools/pull_client.py` | 平台侧多节点拉取客户端 |

## 8. 一键速查（初始化）

```bash
# —— 服务端 ——
tar -xzf ykvos-server-dev-1.0.6.tar.gz && cd release/dev
cp .env.example .env && vi .env
docker compose up -d
# 库已存在时补建 VOS 数据：
docker compose exec mysql bash /init/init-mysql-vos.sh

# —— 每台 VOS 节点 ——
tar -xzf ykvos-agent-1.0.6.tar.gz && cd ykvos-agent
sudo bash install.sh
vi /opt/ykvos-agent/.env          # 写 YK_MYSQL_PWD / YK_SERVER_TOKEN
vi /opt/ykvos-agent/etc/config.yaml  # instance.id / sync.mode / kafka.brokers
/etc/init.d/ykvos-agent restart
```
